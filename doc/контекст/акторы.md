## Краткое описание

В платформе гибких льгот участвуют 10 акторов: 5 человеческих ролей, 3 системных, 2 внешних. Каждый актор описан через роль, действия, данные и точки взаимодействия.

## Люди

### Сотрудник

**Роль.** Конечный пользователь платформы. Получает, распределяет и потребляет льготы.

**Действия:**
1. Регистрация (ФИО, корп. email, ДР, телефон, пароль, галки согласия)
2. Авторизация (логин + пароль; восстановление пароля)
3. Просмотр витрины льгот на Главной странице
4. Выбор и подключение льгот из Каталога (N продуктов, M категорий — конфигурируемо)
5. Управление баллами: просмотр баланса, категорий, транзакций
6. Подключение родственников к ДМС (wizard: программа → оплата → подтверждение)
7. Оплата апгрейда (удержание из ЗП / карта)
8. Просмотр и скачивание документов (заявления, согласия, полисы)
9. Обращение в поддержку (FAQ + форма с 5 категориями)
10. Заявление на маткапитал (wizard: условия → данные ребёнка → подтверждение)

**Данные.** Профиль (ФИО, email, ДР, телефон, табельный номер), баланс баллов, активные льготы, история транзакций, документы, обращения в поддержку.

**Взаимодействие.** → Платформа льгот (основной канал), → **M07 T0705:** Пэйшлюз через payment-gateway (не напрямую!), → Провайдеры льгот (косвенно, через сквозную авторизацию), → HR-менеджер (обращения, проверка прав).

---

### HR-менеджер

**Роль.** Бизнес-оператор льготных программ. Контролирует права сотрудников, обрабатывает исключения, проверяет заявки.

**Действия:**
1. Проверка права на льготу при обращении сотрудника (ФТ §8.12)
2. Ручное добавление сотрудника в реестр при ошибке в автоматической загрузке
3. Обработка обращений из формы поддержки (категоризация, ответ в 1 рабочий день)
4. Проверка заявок на маткапитал (скан свидетельства, стаж ≥ 6 мес, срок ≤ 90 дней)
5. Уведомление сотрудника о результате проверки
6. Рассылка email-уведомлений (новое начисление, новые льготы, статус заявок)

**Данные.** Реестр сотрудников (права, статус), обращения, заявки на маткапитал, шаблоны уведомлений.

**Взаимодействие.** → Платформа льгот (админ-панель), → Сотрудник (уведомления, ответы на обращения), → Кадровая система (запрос данных при проверке).

---

### Администратор платформы

**Роль.** Технический оператор. Обеспечивает загрузку реестров, мониторинг, выгрузку отчётов.

**Действия:**
1. Ручная загрузка XLSX-реестра (при отсутствии API у кадровой системы)
2. Проверка валидации реестра (ошибки, совпадения, дедупликация email+ДР)
3. Скачивание отчёта об ошибках (при наличии)
4. Выгрузка отчётов по шаблонам в Excel (ежемесячный + по запросу)
5. Настройка периодов открытия распределения льгот
6. Мониторинг метрик (активные пользователи, нагрузка, ошибки)

> **Два варианта импорта сотрудников:**
> - **Вариант А — автоматический:** кадровая система предоставляет REST API → ежедневный pull → валидация → дедупликация (email+ДР) → запись в БД
> - **Вариант Б — ручной:** администратор скачивает XLSX с Канала передачи данных → загружает в платформу → валидация → обработка ошибок

**Данные.** XLSX-реестры, отчёты об ошибках, отчёты по шаблонам, метрики.

**Взаимодействие.** → Платформа льгот (админ-панель), → HR-система (запрос данных через API).

---

### Менеджер каталога

**Роль.** Управляет каталогом льгот: добавление/редактирование карточек, продвижение, сезонные наборы, рекомендации, контент.

**Действия:**
1. CRUD карточек льгот (название, описание, стоимость, условия, иконка, статус)
2. Продвижение льгот: баннер, priority_boost, push-уведомление
3. Создание сезонных наборов (состав из карточек, период, баннер)
4. Настройка рекомендаций: правила контекст + сегмент, debug по userId
5. Управление контентом: FAQ CRUD, баннеры, описания карточек
6. Метрики каталога: просмотры, активации, GMV, конверсия, покрытие по категориям
7. Жизненный цикл карточки: Черновик → Активна → Промо → Скрыта → Завершена
8. Закрытие партнёра: деактивация → уведомление сотрудников → возврат баллов

**Данные.** Карточки льгот, коллекции, правила рекомендаций, контент (FAQ, баннеры), метрики.

**Взаимодействие.** → Платформа льгот (админ-панель ЛК-3), → `billing.BillingService` (direct call), → `integrationclient.IntegrationClient` (gRPC → Integration Proxy, статус провайдера, M16).

---

### Администратор интеграций

> **M16:** Provider-интеграции вынесены в отдельный бинарник `lkfl-integration-proxy` (ADR-035). Управление провайдерами через Integration Proxy (gRPC admin endpoints + admin UI ЛК-3). HR sync и payments остались в монолите.

**Роль.** Управляет внешними интеграциями провайдеров льгот через Integration Proxy: подключение новых провайдеров, мониторинг статусов, настройка endpoint'ов, контроль SLA. HR sync и payments управляются отдельно (монолит).

**Действия:**
1. Подключение новой интеграции провайдера: регистрация адаптера (name, protocol, endpoints, auth_method) → YAML config в `provider-configs/`, адаптер в `integration-proxy/adapters/`, тестирование, публикация
2. Мониторинг здоровья: статус каждого адаптера, error_rate, latency, last_sync, circuit breaker state (admin endpoints proxy)
3. Настройка endpoint'ов: изменение URL, auth credentials, retry policy
4. Управление синхронизацией: manual trigger, расписание, error log
5. Контроль SLA: alert при degrade, escalation
6. Деактивация интеграции: блокировка → уведомление HR → поиск замены

**Данные.** Каталог провайдеров, endpoint'ы, auth credentials, health metrics, sync logs, circuit breaker state.

**Взаимодействие.** → Integration Proxy (admin UI ЛК-3: provider management через proxy), → Prometheus/Grafana (monitor), → HR-менеджер (escalation при отказе провайдера).

## Системы

### Платформа льгот

**Роль.** Ядро системы. Обслуживает весь функционал: UI, биллинг, документооборот, комплаенс, нотификации.

**Действия:**
1. Аутентификация и авторизация пользователей
2. Отображение витрины и каталога льгот (N продуктов, M категорий — конфигурируемо)
3. Биллинговый движок: начисление баллов из 4+ источников, списание в N категориях, управление периодами
4. Генерация юридически значимых документов (заявления, согласия ПДн)
5. Lifecycle-менеджмент согласий (создание → действие → продление → отзыв → удаление)
6. Управление wizard-ами (ДМС, маткапитал)
7. Интеграция с внешними системами: `integrationclient/` (gRPC → Integration Proxy: активация/деактивация льгот, статусы провайдеров, M16 ADR-035), `payments/` (оплата, direct call), `billing/payroll/` (удержание из ЗП), `user/HRSync` (синхронизация HR-данных).
8. Система уведомлений (in-app, email, push на мобильном)
9. Поддержка (FAQ, форма обращений, роутинг, SLA)
10. Аналитика и отчёты (метрики, конверсия, выгрузка Excel)
11. White-label конфигурация (тема, логотип, название, каталог)

**Данные.** Пользователи, льготы, транзакции, документы, согласия, обращения, метрики.

**Взаимодействие.** → все остальные акторы.

> **Примечание M12:** актор «Платформа льгот» — абстракция бизнес-процесса. На уровне архитектуры:
> **два бинарника** (`lkfl-server` монолит + `lkfl-integration-proxy`), один `go.mod`, 16 internal-пакетов монолита (14 business + `tenant/` + `api/` + `integrationclient/`).
> Межмодульная коммуникация в монолите — Go interfaces (compile-time type safety). Монолит ↔ proxy — gRPC (localhost). NATS не используется.
> [M12: ADR-024] [M16: ADR-035]
> Полный список пакетов с публичным API → [`пакеты-platform.md`](../архитектура/пакеты-platform.md).

---

### Keycloak — Identity Provider

**Роль.** Центральный компонент безопасности. Единая точка входа для всей платформы. Управляет аутентификацией, авторизацией и сквозным доступом к внешним сервисам.

**Действия:**
1. Аутентификация всех пользователей (OIDC login, refresh, logout, password reset)
2. Эмитирование JWT-токенов с ролями и tenant_id для всех сервисов
3. СSO на внешние сервисы через Identity Broker (Keycloak ↔ провайдер, без участия платформы)
4. Multi-tenancy: realm per tenant (или group-based), изоляция пользователей между компаниями
5. Управление ролями: `employee`, `hr`, `catalog_manager`, `admin`, `integration_admin`
6. Password policy enforcement: сложность, история, ротация (90 дней), блокировка (5 неудач → 15 мин)
7. Audit trail: все события аутентификации логируются

**Данные.** Users (связь с Platform по email), realms, clients, roles, groups, sessions, authentication events, identity broker config.

**Взаимодействие.** → Platform/Billing/Integrations (JWT validation via `go-oidc`), → Frontend (Keycloak JS adapter), → Внешние сервисы (Identity Broker: OIDC/SAML redirect).

---

### Кадровая система

**Роль.** Источник истины о сотрудниках. Содержит актуальный реестр сотрудников, их грейды, стаж, статусы (активный / уволен).

**Действия:**
1. Ежедневная синхронизация данных о сотрудниках через REST API (ФИО, email, ДР, телефон, пол, адрес, стаж, грейд, статус)
2. REST API endpoint для синхронизации (pull по расписанию)

**Данные.** Реестр сотрудников (ФИО, email, ДР, телефон, пол, адрес ФИАС, табельный номер, стаж, грейд, статус).

**Взаимодействие.** → Платформа `user/HRSync` (REST API pull ежедневно, Asynq worker, M11 T1102: bypasses hub).

## Внешние

### Провайдеры льгот

**Роль.** Исполнение льготных обязательств. Предоставляют конкретные услуги (ДМС, фитнес, питание, обучение, мерч, связь) на условиях платформы.

**Действия:**
1. Активация льготы для сотрудника (gRPC `Activate()` через Integration Proxy, M16 ADR-035)
2. Деактивация льготы (gRPC `Deactivate()` через Integration Proxy, M16 ADR-035)
3. Передача статусов (активен / истёк / отозван)
4. Предоставление данных для полисов (ДМС), сертификатов (фитнес)
5. Сквозная авторизация сотрудника (без повторного входа)

**Данные.** Активации, статусы, полисы, сертификаты.

**Провайдеры из прототипа:** АльфаСтрахование, World Class, СберСпорт, Яндекс Еда, Skillbox, Яндекс Психотерапия, СДЭК Store, Мать и дитя, Skyeng, МТС, Ready4, Подарок в квадрате, PrimeZone — текущий список. Количество не фиксировано — «сколько есть, столько и будет».

**Взаимодействие.** ← Integration Proxy `integration-proxy/adapters/` (gRPC от монолита через `integrationclient/`, M16 ADR-035), → Сотрудник (косвенно, через сквозную авторизацию).

### Пэйшлюз / 1С

**Роль.** Обработка платежей. Два канала оплаты сверх базовых льгот.

**Действия:**

| Канал | Действие | Источник |
|-------|----------|----------|
| Пэйшлюз | Авторизация карты (Visa/MC/МИР), заморозка средств, подтверждение платежа | ФТ §8.9, прототип DMS шаг 2 |
| 1С | Приём заявления на удержание из ЗП, расписание на 3 месяца равными частями | прототип DMS шаг 2, FAQ |

**Данные.** Транзакции оплаты, заявления на удержание, статусы платежей.

**Взаимодействие.** ← `internal/payments/` (direct call, M12: был Integrations Hub), ← `billing/payroll/` (direct call, M11 T1103: был Integrations Hub).

## Матрица взаимодействий

> **M12:** lkfl-server — один бинарник, 17 internal-пакетов (исторически). Межмодульная коммуникация — Go interfaces (compile-time type safety). NATS не используется.
> **M16:** lkfl-server + lkfl-integration-proxy — два бинарника, 16 internal-пакетов монолита. Монолит ↔ proxy — gRPC (localhost). [ADR-035]
