# Философия платформы

> **Этот документ читать перед остальной документацией.**

---

## Что это

LKFL — **универсальная white-label платформа корпоративных льгот**. Одна кодовая база. Любой бренд. Любой набор льгот. Любая модель начислений. Любые активности.

СДЭК — **первый tenant**, референс для валидации. НЕ целевой заказчик в архитектурном смысле.

---

## Три нуля

| Принцип | Что значит | Как обеспечено |
|---------|-----------|----------------|
| **Нулевая привязка к бренду** | Новый tenant без изменения кода — только CSS + конфиг | Multi-tenancy, CSS-переменные, realm per tenant, brand config |
| **Нулевая привязка к льготам** | Новый провайдер — конфигурация, не код | ProviderAdapter interface (Integration Proxy), CRUD через ЛК-3 |
| **Нулевая привязка к модели начислений** | Правила биллинга конфигурируются, не программируются | Billing Rule Engine: формулы, условия, приоритеты, триггеры |

---

## Не иерархия. Платформа.

Платформа **не построена вокруг конкретного набора льгот конкретного бренда**. Она построена вокруг абстракций:

- **Льгота** — не «ДМС от Альфы» и не «фитнес World Class». Льгота = карточка с названием, стоимостью, категорией, провайдером и правилами подключения. Конкретные примеры — иллюстрации.
- **Баллы** — не обязательно баллы. Могут быть рубли, проценты, кредиты. Валюта — конфиг.
- **Активности** — не «опрос» и «событие». Активность = тип с критериями завершения и вознаграждением. Типы — CRUD.
- **Категории** — не «ДМС, Спорт, Питание». M категорий — конфигурируемо.

## СДЭК в документации

Примеры в документации используют данные СДЭК для конкретики. Это **иллюстрации**, не ограничения.

| Где СДЭК-данные | Как читать |
|----------------|-----------|
| Таблицы с ценами льгот | Пример tenant «СДЭК». Другой tenant — другие цены, другие продукты |
| Список провайдеров (Альфа, World Class...) | Пример адаптеров. Новый провайдер = новый адаптер + конфиг |
| Цвета `#00B33C`, логотип СДЭК | Пример brand config. Другой tenant — другой CSS |
| Маткапитал 100 000₽ | Пример welfare-программы. Другой tenant — другие программы |

## Как добавить новую компанию (кратко)

1. Создать tenant → Keycloak realm → CSS → seed-данные
2. Настроить Billing Rules: источники начислений, категории, формулы
3. Подключить провайдеров: выбрать адаптеры, ввести endpoint'ы
4. Заполнить каталог: категории, карточки льгот, сезонные наборы
5. Настроить активности: типы, правила, целевые аудитории
6. Импорт сотрудников: XLSX или API
7. Тестовый прогон

Подробный walkthrough → `спецификация/journeys/tenant-onboarding.md`.

## Что мы НЕ делаем без изменения кода

| Что | Почему требует код |
|-----|-------------------|
| Новый протокол интеграции (SAML/SOAP/бинарный) | ProviderAdapter — Go interface, нужен новый адаптер |
| Новый метод аутентификации | Keycloak OIDC — протокол фиксирован |
| Новый тип базы данных | PostgreSQL — архитектурное решение |

> **M07 T0704 (ADR-017):** REST-провайдер — без кода! YAML config + Go-шаблон. Новый REST-провайдер = YAML файл.

Всё остальное — конфигурация.
