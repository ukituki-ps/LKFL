## Краткое описание

Платформа гибких льгот работает в жёстких нефункциональных ограничениях: 100 000+ пользователей, ФСТЭК-сертификация, 152-ФЗ, жёсткие сроки запуска. Каждое ограничение диктует архитектурные решения — выбор стека, topology, стратегию хранения данных.

## Подробный разбор

### 1. Масштаб — 100 000+ активных пользователей

**Что означает.** Пик нагрузки: начало периода распределения льгот (сотни тысяч обращений за несколько дней). Регулярная нагрузка: ежедневные визиты сотрудников (проверка баланса, подключение льгот).

**Требования:**
- Горизонтальное масштабирование (добавление инстансов без изменения кода)
- Stateless-архитектура application layer (возможность распределения запросов на любой инстанс)
- Эффективные запросы к БД (индексация, кэширование)
- Rate limiting на чувствительные endpoints

**Источники.** ФТ §5 — «100 000+ активных пользователей, горизонтальное масштабирование».

---

### 2. Безопасность — OWASP, шифрование, валидация, сканирование

**Требования:**
- OWASP Top 10 защита (XSS, CSRF, SQL injection, и т.д.)
- Шифрование данных в покое (БД, файлы) и при передаче (TLS 1.2+)
- Валидация паролей (сложность, история, ротация)
- Регулярное сканирование уязвимостей (SAST/DAST)

**Требования к паролям.** Из прототипа: пароль на регистрации. ФТ §8.1 не детализирует политику паролей → определить на этапе архитектуры.

**Источники.** ФТ §5 — «OWASP, шифрование данных, валидация паролей, сканирование уязвимостей».

---

### 3. Отказоустойчивость — HA-архитектура

**Требования:**
- Резервирование БД (master-slave, replication)
- Балансировка нагрузки (load balancer перед application instances)
- Stateless application layer (Keycloak OIDC — JWT валидация на каждом инстансе; Redis — кэш токенов, не session store)
  > ФТ §5 упоминает «sticky sessions», но архитектура **отказалась** от них в пользу stateless JWT + Redis cache. Это упрощает балансировку и горизонтальное масштабирование.
- Кластерная архитектура (мульт-инстанс deployment)
- Health-check endpoints для мониторинга

**Источники.** ФТ §5 — «Резервирование БД, балансировка нагрузки, sticky sessions, кластерная архитектура».

---

### 4. Комплаенс — ФСТЭК, 152-ФЗ, единое место хранения

**Требования:**
- ФСТЭК-сертификация платформы (процесс сертификации → учёт сроков)
- 152-ФЗ: персональные данные не покидают территорию РФ
- Единое место хранения данных (не разбрасывать ПДн по внешним системам)
- Audit trail: кто, когда, какие данные доступался — для проверки

**Сложности.** ФСТЭК-сертификация — не «настроили и забыли». Это процесс, который требует: описание архитектуры, политики безопасности, penetration testing, документирование всех компонентов.

**Источники.** ФТ §5 — «ФСТЭК-сертификация, единое место хранения данных, 152-ФЗ». ФТ §8.1 — «ПДн: без галки согласия — нельзя передавать данные сторонним компаниям».

---

### 5. ПДн — lifecycle согласий

**Требования:**
- Без галки согласия — нельзя передавать данные сторонним компаниям (в т.ч. по API)
- При письменном обращении — снять галку, удалить данные
- Дети до 18 лет — требуется согласие родителя на передачу ПДн
- Согласие родственников 18+ — через SMS-код
- Хранение: полисы и документы — в ядре системы, на платформе — только по запросу

**Архитектурное влияние:**
- Separate consent store с audit trail
- Data deletion pipeline (при отзыве согласия — каскадное удаление)
- Parental consent workflow для детей < 18

**Источники.** ФТ §8.1, §8.13.

---

### 6. Браузерная поддержка

**Требования:**
- Desktop: Firefox, Safari, Chrome
- Mobile: iOS Safari, Android Chrome
- Responsive дизайн (viewport, adaptive layout)
- Mobile-first подход (см. проблему 12)

> **Примечание:** ФТ §5 упоминают IE и Opera. IE удалён — browser terminated (2022). Opera — 0.03% рынка; поддержка покрывается через стандарты (Chromium-based).

**Источники.** ФТ §5 — «Браузеры: Firefox, Opera, Safari, Chrome; Mobile: iOS Safari, Android Chrome».

### 7. Сроки — лето 2026

**Требования:**
- Запуск решения — лето 2026
- Полная работа — август 2026

**Архитектурное влияние.** MVP-фазировка:
- Этап 1: базовый портал + ДМС + баллы + каталог
- Этап 2: аналитика, метрики, почтовая рассылка (ФТ §8.11)
- Этап 3: доплата сверх баланса, расширенные интеграции (ФТ §8.13)

**Источники.** Кубики §Таймлайн — «Запуск решения: Лето 2026. Полная работа: Август 2026».

### 8. Система нотификаций

**Требования:**
- Уведомления для пользователей (начисление, новые льготы, статус заявок)
- Уведомления для администраторов (ошибки загрузки, критические события)
- Каналы: email (существующий), in-app (прототип показывает), push (мобильный)

**Источники.** ФТ §5 — «Система нотификаций для пользователей и администраторов». Процесс регистрации §5, шаг 9 — «Рассылка по email со ссылкой на платформу». Прототип — лента событий на Dashboard.

### 9. Журналирование

**Требования:**
- Logging всех событий и операций
- Audit trail для ПДн и согласий (кто, когда, что)
- Структурированные логи (machine-readable)
- Centralized log aggregation (для анализа и мониторинга)

**Источники.** ФТ §5 — «Logging всех событий и операций».

---

### 10. Интеграции

**Требования:**
- Кадровые системы (выгрузка реестра, синхронизация статусов)
- Провайдеры льгот (активация/деактивация, статусы)
- Пэйшлюз (онлайн-оплата)
- 1С (удержание из ЗП)
- Внешние сервисы (digest.ready4.ru, legal.ready4.ru, www.ready4.ru, Подарок в квадрате, PrimeZone) — сквозная авторизация

**Архитектурное влияние.** API gateway для маршрутизации внешних вызовов. Message broker для асинхронного обмена с провайдерами. Retry logic с exponential backoff. Circuit breaker для отказов внешних сервисов.

**Источники.** ФТ §6.2 — «Передача данных из/в систему Заказчика». ФТ §6.3 — «Сквозная авторизация на сторонних сервисах». ФТ §8.4–8.8 — 5 внешних сервисов.

### 11. Геолокация — J39

**Требования:**
- Browser Geolocation API для определения местоположения сотрудника на мобильных устройствах
- Push-уведомление при входе в геозону клиники сети ДМС (J39)
- Карта клиник с маркерами ближайших точек (S10, J39)
- Кнопка «Навигация» → открывает Maps-приложение

**NFR:**
- **152-ФЗ:** геоданные — персональные данные. Требуется отдельное согласие ПДн на обработку геолокации
- **Permissions-Policy:** `geolocation=()` по умолчанию; включается только на страницах с клиникой (S10)
- **Mobile-only:** geolocation не используется на desktop (не поддерживается стабильно)
- **Opt-in:** сотрудник явно разрешает геолокацию через browser prompt

**Источники.** J39 — «Мобильный push: геолокация клиник рядом». `архитектура/безопасность.md § Permissions-Policy`.
