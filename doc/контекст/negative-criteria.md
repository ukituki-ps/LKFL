## Краткое описание

Negative criteria исключают подходы и решения, которые не соответствуют целям проекта. Это не ограничения — это осознанные решения «не делать», основанные на анализе артефактов, прототипа и бизнес-контекста.

## Подробный разбор

### 1. Не строим monolith с монолитным фронтом

**Почему.** Прототип — SPA с 5 страницами и 3 модалками, разделёнными по роутам (`#page-dashboard`, `#page-catalog` и т.д.). Архитектура платформы — модульная: UI, биллинг, документооборот, интеграции — отдельные компоненты.

**Что вместо этого.** Модульная архитектура с чёткими границами:
- Frontend — SPA (React или аналог)
- Backend — API server (Go)
- Биллинг — отдельный модуль
- Интеграции — adapter pattern

---

### 2. Не храним полисы и документы на платформе

**Почему.** ФТ §8.13: «Хранение: полисы и документы — в ядре системы, на платформе — только по запросу». Платформа — витрина и интерфейс. S3-compatible storage — хранилище файлов.

**Что вместо этого.** Платформа хранит метаданные документов (название, тип, статус, дата, `storage_path`). Генерация PDF — на лету (из шаблонов). Хранение сгенерированных файлов — S3-compatible storage. При запросе «Скачать» — получение файла из storage, передача пользователю.

**Источники.** ФТ §8.13 — «Хранение: полисы и документы — в ядре системы, на платформе — только по запросу». `безопасность.md` → Backup: S3-compatible storage + replication.

---

### 3. Не обрабатываем ПДн без явного согласия

**Почему.** 152-ФЗ + ФСТЭК. Прямое требование ФТ §8.1: «Без галки согласия — нельзя передавать данные сторонним компаниям, в т.ч. по API». При письменном обращении — удалить данные.

**Что вместо этого.** Consent-first архитектура:
- Каждый документ согласия — отдельная запись с timestamp, scope, status
- API-запросы к провайдерам проверяют наличие активного согласия
- Отзыв согласия → каскадное удаление данных из внешних систем

**Источники.** ФТ §8.1 — «Без галки согласия — нельзя передавать данные сторонним компаниям. Запрет обработки: при письменном обращении — удалить данные».

---

### 4. Не привязываемся к одному бренду

**Почему.** Прототип закреплён СДЭК (навигация «СДЭК · Льготы», цвет `#00B33C`, текст «сотрудник СДЭК»), но это референс. Платформа — white-label: один код, любой бренд.

**Что вместо этого.** CSS-переменные для темы, конфигурация бренда (логотип, название, цвета), тексты-шаблоны (`${company.name}`), каталог льгот как подключаемые модули.

**Источники.** Прототип: `<title>СДЭК · Льготы</title>`, `--green: #00B33C`, `<div>сотрудник СДЭК</div>`. Проблема 11.

---

### 5. Не используем ручной документооборот

**Почему.** 12 проблем из `проблема.md` показывают, что ручной документооборот — одна из главных болей. Без digital-документооборота:
- HR не видит статусы согласий на 100 000+ сотрудников
- Юридический отдел не проходит проверки ФСТЭК
- Сотрудник не скачивает полис

**Что вместо этого.** Digital lifecycle для всех документов:
- Генерация шаблонов (заявление, согласие)
- Электронное подписание (галка в wizard-е)
- Статусы (Ожидает → Подписано → Активен → Истёк)
- Скачивание PDF по запросу
- Автоматическое уведомление о предстоящем окончании

**Источники.** Прототип: страница «Документы», DMS Wizard шаг 3, Mat Capital Wizard шаг 3. Проблема 7.

---

### 6. Не интегрируем платформу напрямую с кадровой системой

**Почему.** Прямая точка-в-точку интеграция → тесная связанность, зависимость от downtime кадровой системы, сложность отладки, нарушение архитектуры единого шлюза.

**Что вместо этого.** HR синхронизация → `internal/user/HRSync` direct REST pull (M11 T1102: bypasses Integrations Hub, был NATS):
- HR-система предоставляет REST API endpoint
- Platform `user/HRSync` делает pull по расписанию (Asynq worker `hr-sync-daily`, ежедневно)
- Результат → валидация → дедупликация (email + ДР) → запись в БД напрямую
- Валидация → дедупликация (email + ДР) → запись в БД
- При ошибках → alert администратору → ручная коррекция
- Обмен асинхронный, с автоматическим retry
- HR-менеджер не обращается к кадровой системе напрямую — Platform pull по расписанию (Asynq worker)

**Источники.** Процесс регистрации §5 — синхронизация реестра сотрудников.

---

### 7. Не строим монолитную БД без репликации

**Почему.** ФТ §5 требует: «Резервирование БД, балансировка нагрузки, кластерная архитектура». 100 000+ пользователей ≠ single DB. Stateless архитектура (JWT + Redis) исключает необходимость sticky sessions.

**Что вместо этого.** Master-slave replication, read replicas для аналитики, connection pooling, failover strategy.

---

### 8. Не строим разовые порталы под одну компанию

**Почему.** Каждая компания строит свой портал (СДЭК → свой, Заказчик → свой, следующая → свой). Это не масштабируемо. Наша цель — universal white-label платформа.

**Что вместо этого.** Multi-tenant архитектура:
- Один код
- Конфигурация на уровне tenant (бренд, каталог льгот, тема, правила)
- Изоляция данных между tenants
- Возможность добавления нового клиента без переписывания кода

---

### 9. Не делаем прямые вызовы platform → внешний мир

**Почему.** Прямые вызовы создают тесную связанность, зависят от downtime внешних систем, нарушают ПДн (нет проверки consent), усложняют аудит. Блокировка горутин в hot path при прямых HTTP calls к внешним провайдерам (ADR-035).

**Что вместо этого.** Все вызовы провайдеров льгот через Integration Proxy (`lkfl-integration-proxy`, gRPC, M16 ADR-035):
- Монолит `integrationclient.IntegrationClient` → gRPC → Integration Proxy → HTTP call к провайдеру
- Activate/Deactivate — асинхронные (Proxy возвращает `job_id`, результат через webhook callback)
- GetProviderStatus/GetCatalog — синхронные (быстрые < 5s)
- Circuit breaker per provider — на уровне Proxy процесса
- Исключение: SSO browser redirect через Keycloak Identity Broker (платформа не участвует)
- Исключение: HR sync → `internal/user/HRSync` direct REST pull (M11 T1102, не затронут)
- Исключение: Payments → `internal/payments/` direct REST (M11 T1103, не затронут)
- Исключение: 1C payroll → `internal/billing/payroll/` direct REST (M11 T1103, не затронут)
- Локальный кэш каталога в PostgreSQL — fallback при недоступности Proxy

---

### 10. Не эмитим собственные токены аутентификации

**Почему.** Собственные токены дублируют Keycloak, нарушают единую точку входа, усложняют SSO, не покрывают ФСТЭК-комплаенс.

**Что вместо этого.** Keycloak — единственный источник токенов:
- Все сервисы валидируют JWT от Keycloak (`go-oidc`)
- Platform не эмитит токены, не хранит пароли
- Token cache в Redis (DB 0) — кэш результатов JWT-верификации (signature валидируется локально по JWKS; Redis хранит userId→roles на TTL токена)
- SSO на внешние сервисы — Keycloak Identity Broker (OIDC/SAML redirect)
