### J15 — Открытие периода распределения

**Триггер:** начался новый период.
**Потребность:** открыть окно распределения льгот для сотрудников.
**Артефакты:** H02, H05

**Путь:**
1. HR входит в админ-панель → «Управление периодами» (H02)
2. Устанавливает дату открытия и закрытия распределения (H02)
3. Система отправляет уведомления всем активным сотрудникам (H02)
4. Сотрудники получают доступ к распределению (S03)
5. HR следит за метриками в реальном времени (H02 → metrics)
6. HR видит общий прогресс в аналитике (H05)

**Exit criteria:** период открыт, сотрудники уведомлены, метрики видны.

### J16 — Подключение нового сотрудника

**Триггер:** в компанию принят новый сотрудник.
**Потребность:** добавить его в систему льгот.
**Артефакты:** H01

**Путь:**
1. HR-система синхронизирует данные через REST API: ФИО, email, ДР, телефон, пол, адрес, статус (H01)
2. Платформа валидирует поля (H01 → validation):
   - Обязательные поля
   - Дубликаты по email + ДР
   - Формат данных
3. При ошибках: alert администратору → ручная коррекция (H01 → error_report)
4. При успехе: сотрудник добавлен → получает email-уведомление (H01)
5. Сотрудник регистрируется → проходит J01 (S01)

**Exit criteria:** сотрудник синхронизирован из HR-системы, получил email, зарегистрирован.

### J17 — Увольнение: отключение сотрудника

**Триггер:** сотрудник уволился.
**Потребность:** отключить доступ, открепить от программ, сформировать реестры.
**Артефакты:** H01, H05

**Путь:**
1. HR обновляет реестр: статус → «Деактивирован» → загружает (H01)
2. Система выполняет каскадную деактивацию (T04):
   - ① Блокировка доступа к ЛК
   - ② Открепление от льгот (сотрудник + родственники)
   - ③ Остановка billing rules
   - ④ Запрос баланса партнёрским сервисам
   - ⑤ Формирование реестра открепления ДМС
3. HR выгружает отчёт по отключённому сотруднику (H05)

**Exit criteria:** доступ отключён, реестры сформированы.

### J18 — Стимулирование целевых действий (геймификация, type=activity engagements)

**Триггер:** HR хочет повысить активность сотрудников.
**Потребность:** создать activity-энгейджмент → сотрудник выполняет → получает баллы (credit) → использует benefit-энгейджменты.
**Артефакты:** H03, H05, B01

**Путь:**
1. HR создаёт activity-энгейджмент: название, описание, engagement type (опрос / событие / реферал / custom) → `POST /admin/engagements?type=activity` (H03)
2. Назначает бонусные баллы → привязка к Billing Rule (B01, billing_direction=credit) (H03)
3. Устанавливает период действия, eligibility (заменяет старый target_audience) (H03)
4. Настраивает уведомления: email / push / лента событий (H03 → notification)
5. Activity опубликована → сотрудники видят в Каталоге `type=activity` (S04)
6. Сотрудники выполняют через Engagement Wizard → получают баллы (J12)
7. HR следит за метриками `GET /admin/engagements/analytics` (H03 → metrics):
   - % получивших уведомление
   - % выполнивших
   - Начисленные баллы
   - Конверсия «получил баллы → потратил на benefit»

**Exit criteria:** activity-энгейджмент создан, сотрудники участвуют, метрики собираются.

### J19 — Анализ эффективности программы

**Триггер:** HR хочет понять, как работает программа льгот.
**Потребность:** отчёты, метрики, инсайты.
**Артефакты:** H05, H01

**Путь:**
1. HR входит в админ-панель → «Аналитика» (H05)
2. Видит дашборд (H05 → dashboard):
   - Заявленные / зарегистрированные / распределившие
   - Популярные льготы (по количеству активаций)
   - Общий расход баллов по категориям
   - Конверсия «баланс → расход»
3. Выгружает ежемесячный отчёт (H05 → report_types):
   - Excel, автоматический, хранение N месяцев
4. Формирует отчёт по запросу (H05 → user_search):
   - Поиск по ФИО+ДР → детали сотрудника (H01)

**Exit criteria:** HR получил отчёты, видит метрики, принимает решения.

### J20 — Апрув заявок сотрудников

**Триггер:** сотрудник подал заявку, требующую проверки.
**Потребность:** проверить документы и одобрить/отклонить.
**Артефакты:** H04

**Путь:**
1. HR видит заявку в списке (H04 → request_list):
   - Маткапитал (из J06)
   - Психолог (из J07)
   - Апгрейд ДМС с удержанием из ЗП (из J05)
2. Проверяет документы: свидетельства, стаж, согласия (H04 → request_item)
3. **Одобрение** (H04 → approve_action):
   - Статус → Активна
   - Сотрудник получает уведомление
   - Льгота/выплата активируется
   - Billing Rule выполняется (B01)
4. **Отклонение** (H04 → reject_action):
   - Статус → Отклонена
   - Сотрудник получает уведомление с причиной
   - Льгота не активируется
   - Баллы возвращены (B01)

**Exit criteria:** заявка обработана, сотрудник уведомлён.

### J20a — Настройка ежемесячных начислений

**Триггер:** нужно изменить правила начисления (новый грейд, изменение тарифа).
**Потребность:** обновить таблицу грейдов → баллов.
**Артефакты:** H03, B01

**Путь:**
1. HR открывает «Геймификация» → «Правила начисления» (H03 → engagement-types)
2. Редактирует таблицу грейд → баллы (B01):
   - Добавляет новый грейд
   - Изменяет сумму для существующего
   - Устанавливает период вступления в силу
3. Сохраняет → правило обновлено (B01)
4. Следующее начисление (T01) использует новые правила

**Exit criteria:** правила начисления обновлены, применяются с указанной даты.

### J32 — Массовое отправление уведомлений

**Триггер:** HR нужно отправить broadcast-уведомление всем сотрудникам или сегменту.
**Потребность:** единый модуль email/SMS/push рассылок.
**Артефакты:** H06, B01, S03

**Путь:**
1. HR открывает H06 «Уведомления» → выбирает тип: email / SMS / push / все
2. Выбирает аудиторию: все активные / сегмент (грейд, отдел, статус) / список импортируемый из Excel
3. Заполняет шаблон: тема, текст (white-label из J44), прикреплённая ссылка
4. Выбирает время: «Сейчас» / «Запланировано» (дата + время)
5. Система отправляет → H06 показывает метрики: отправлены, доставлены, открыты, клики
6. ЛК-1 получает push в колокольчик (S03)
7. Audit-лог (H07) фиксирует рассылку: кто, кому, когда, сколько

**Exit criteria:** уведомление доставлено целевой аудитории, метрики доставки собраны.

### J37 — Аудит-лог действий администраторов

**Триггер:** compliance-требования (ФСТЭК, 152-ФЗ) + необходимость отследить, кто/что изменил.
**Потребность:** immutable-лог всех admin-действий.
**Артефакты:** H07, H01, H02, H03, H04, H05

**Путь:**
1. Каждое CRUD-действие администратора (H01–H05, H06) автоматически логируется
2. Запись: timestamp, admin_id, action_type (create/update/delete), resource_type, resource_id, old_value, new_value, ip_address
3. HR с ролью `admin` видит таблицу audit-логов с фильтрами: по пользователю, по ресурсу, по периоду, по действию
4. Export в Excel → для внешних аудиторов
5. Логи immutable: DELETE не разрешён, только append

**Exit criteria:** все admin-действия логгируются, доступны для фильтрации и выгрузки, данные неизменяемы.

### J38 — Экспорт/импорт конфигурации tenant'а

**Триггер:** нужна миграция staging→production или создание тестовой копии tenant'а.
**Потребность:** backup/restore всей конфигурации.
**Артефакты:** H08, B01, M01–M07

**Путь:**
1. Admin (`admin` role) открывает H08 → нажимает «Экспорт конфигурации»
2. Система формирует JSON-архив: billing rules (B01), engagement types (M01), offers, flows, collections (M04), recommendations (M06), content (M07), periods (H02)
3. Архив скачивается на диск
4. Import: admin загружает архив → система валидирует → применяет конфигурацию
5. Валидация: конфликтующие ID → ремап, отсутствующие зависимости → error_report
6. Audit-лог (H07) фиксирует весь import/export

**Exit criteria:** конфигурация экспортирована/импортирована, integrity проверена, аудит зафиксирован.

### J40 — Escalation rules для заявок на апрув

**Триггер:** HR загружен / заявка превышает порог суммы / требует высшего аппрува.
**Потребность:** автоматическая маршрутизация заявок по уровням.
**Артефакты:** H04 (расширение), H05, B01

**Путь:**
1. Admin настраивает escalation rules в H04:
   - По сумме: > X баллов → требует аппрув руководителя
   - По типу: маткапитал → HR Lead, психолог → HR Specialist
   - По load: > N ожидающих у HR → auto-escalate к коллеге
2. Заявка поступает → система проверяет правила → назначает ответственного
3. Если ответственный не обработал за T часов → auto-escalate на уровень выше
4. Все escalation-события логгируются в H07 (audit-лог)
5. HR видит escalation-chain в деталях заявки
6. Метрики H05: avg_time_to_approve, escalation_rate, SLA breaches

**Exit criteria:** заявка маршрутизирована по правилам escalation, обработана в SLA, цепочка задокументирована.
