# Системные триггеры

## TL;DR (для агентов)

> 7 автоматических триггеров (по расписанию или событию). 114 строк.
> - **Связь с биллингом:** каждый триггер → Billing Rule (B01)
> - **Полный список:** строка 10

Автоматические процессы, не требующие действий пользователя. Выполняются системой по расписанию или событию.

**Связь с биллингом:** каждый триггер активирует Billing Rule (B01).
**Логging:** все триггеры записываются в H07 (audit-лог).

---

### T01 — Ежемесячное начисление баллов

**Триггер:** 1-е число каждого месяца, cron.
**Артефакты:** B01, S03, S05
**Billing Rule:** direction=credit, frequency=monthly, trigger=cron

**Действие:**
1. Система проходит всех активных сотрудников (H01)
2. Для каждого: lookup грейд → таблица грейд→баллы (B01)
3. Создание транзакции: credit, amount=баллы, category=Общая (B01)
4. Обновление счёта сотрудника (S05)
5. Лента событий: «Ежемесячное начисление +N баллов» (S03)

---

### T02 — Напоминание о сгорании баллов

**Триггер:** N дней до конца периода распределения (пример: 30 дней).
**Артефакты:** S03, S05

**Действие:**
1. Система находит сотрудников с остатком > 0 (S05)
2. Рассчитывает дни_до_сгорания < N
3. Отправляет email-уведомление
4. Отправляет push в ЛК: «Ваши баллы сгорят через N дней» (S03)
5. Активирует предупреждение «Пора потратить» в S05

---

### T03 — Бонус за юбилейный стаж

**Триггер:** дата достижения стажа (пример: 3 года).
**Артефакты:** B01, S03, S05
**Billing Rule:** direction=credit, frequency=one-time, trigger=cron

**Действие:**
1. Система проверяет стаж сотрудников (H01)
2. Для достигших юбилея: создание транзакции (B01)
3. Обновление счёта (S05)
4. Лента событий: «Бонус за стаж N лет» (S03)

---

### T04 — Каскадная деактивация

**Триггер:** сотрудник деактивирован в реестре (J17).
**Артефакты:** H01, S02, B01

**Действие:**
1. Блокировка доступа к ЛК (S02)
2. Открепление от льгот (сотрудник + родственники)
3. Отмена активных Billing Rule (B01):
   - Остановка ежемесячных списаний
   - Возврат остатка за период
4. Запрос баланса партнёрским сервисам (E01)
5. Формирование реестра открепления (H01)

---

### T05 — Периодическое списание льгот

**Триггер:** 1-е число месяца (для льгот с frequency=monthly).
**Артефакты:** B01, S03, S05

**Действие:**
1. Система проходит все активные льготы сотрудников
2. Для льгот с frequency=monthly: выполнение Billing Rule (B01)
3. Создание транзакции: debit, amount=стоимость (B01)
4. Обновление счёта (S05)
5. При нехватке: отмена льготы + уведомление (S03)

---

### T06 — Ежедневный health-check всех интеграций

**Триггер:** cron, ежедневно (02:00). Дополняет manual health из T0708.
**Артефакты:** B01, S03
**Logging:** H07 (audit-лог)

**Действие:**
1. Integration Proxy выполняет health-check всех активных провайдеров
2. Для каждого: ping endpoint, проверить error_rate_30d, latency_p95
3. При деградации (error_rate > 5% или latency > 2с):
   - Alert Integration Admin через H06 (email + push)
   - Автоматический retry с fallback-провайдером (если настроен)
4. Результаты записываются в SLA dashboard (T0708)
5. Audit-лог (H07): timestamp, provider_id, status, error_rate, latency

---

### T07 — Автоматическая ротация секретов провайдеров

**Триггер:** cron, по расписанию (ежемесячно или при истечении TTL секрета).
**Артефакты:** B01
**Logging:** H07 (audit-лог)

**Действие:**
1. Система определяет секреты, требующие ротации (API keys, bearer tokens, OAuth refresh tokens)
2. Для каждого провайдера с поддержкой ротации:
   - Запрашивает новые credentials через provider API
   - Валидирует новые credentials (test transaction)
   - Атомарная замена: write new → verify → delete old
3. При неудаче ротации: alert Integration Admin, повтор через N часов (max 3 попытки)
4. Audit-лог (H07): provider_id, secret_type, rotation_status, timestamp
5. Секреты хранятся в зашифрованном виде (HSM / KMS) — не логгируются
