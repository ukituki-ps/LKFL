### J21 — Добавление нового партнёра (type=benefit engagement)

**Триггер:** пришёл новый провайдер с предложением.
**Потребность:** интегрировать partner как benefit-энгейджмент (debit) в каталог.
**Артефакты:** M01, M02, M05, B01

**Путь:**
1. Менеджер оценивает предложение: подходит ли под программу, стоимость, условия
2. Создаёт engagement type (M01): `type="benefit"`, название, описание, provider_adapter, category, иконка → `POST /admin/engagement-types?type=benefit`
3. Создаёт engagement offer: стоимость, eligibility, billing_rule_id (B01, billing_direction=debit), flow_id
4. Создаёт engagement flow (steps: form → approval → provider_api → document_generation)
5. Публикует в каталоге (M05 → available) → бейдж «Новинка»
6. Отслеживает первые активации в метриках (M02, `/admin/engagement-types/analytics`)

**Exit criteria:** partner в каталоге как benefit-энгейджмент, оффер опубликован, активации идут.

---

### J22 — Изменение стоимости или условий benefit-энгейджмента

**Триггер:** провайдер изменил цену или условия.
**Потребность:** обновить engagement offer (benefit).
**Артефакты:** M01, M02

**Путь:**
1. Менеджер открывает engagement offer → редактирует стоимость/eligibility → `PUT /admin/engagements/:id` (M01)
2. Обновляет Billing Rule: новая сумма (B01)
3. Сохраняет → offer обновляется в каталоге (S04)
4. Уже подключенные сотрудники: уведомление об изменении (если критично) (S03)
5. Новые сотрудники: видят обновлённый offer
6. Метрики обновляются: новая стоимость отражена в GMV (M02)

**Exit criteria:** offer обновлен, сотрудники уведомлены при необходимости.

---

### J23 — Анализ использования benefit-энгейджментов

**Триггер:** менеджер хочет понять, какие benefits работают, а какие нет.
**Потребность:** метрики по каждому benefit-энгейджменту в каталоге.
**Артефакты:** M02

**Путь:**
1. Менеджер открывает дашборд каталога → `GET /admin/engagement-types/analytics` (M02)
2. Видит по каждому benefit-офферу (M02 → per_item_metrics):
   - Просмотры карточки
   - Количество активаций
   - Стоимость (cost.amount)
   - Доля от общего расхода баллов
   - Конверсия просмотр → активация
3. Выявляет проблемные offers: 0 активаций, низкая конверсия (M02)
4. Принимает решение:
   - Доработать описание → (J28)
   - Изменить стоимость → (J22)
   - Продвинуть → (J25)
   - Убрать → (J26)

**Exit criteria:** аналитика собрана, решения приняты.

### J24 — Балансировка каталога benefit-энгейджментов

**Триггер:** в одной категории benefit-энгейджментов только один провайдер.
**Потребность:** обеспечить выбор: 2-3 провайдера на категорию.
**Артефакты:** M02, M01, M05

**Путь:**
1. Менеджер проверяет покрытие по категориям (M02 → per_category_metrics, filter type=benefit)
2. Выявляет пробелы: категории с <2 провайдерами (M02)
3. Ищет новых провайдеров
4. Добавляет партнёра как новый benefit engagement type (J21) → проверяет разнообразие (M02)
5. Периодически проверяет: нет ли монополии одного провайдера

**Exit criteria:** каждая категория benefit-энгейджментов имеет 2-3 провайдера.

---

### J25 — Продвижение benefit-энгейджмента

**Триггер:** менеджер хочет увеличить активации конкретного benefit-энгейджмента.
**Потребность:** привлечь внимание сотрудников.
**Артефакты:** M03, M05, M02

**Путь:**
1. Менеджер выбирает benefit offer для продвижения (M01)
2. Смена status: active → promotional → `PATCH /admin/engagements/:id/status` (M05)
3. Настраивает подсветку (M03):
   - Баннер на Главной (M03 → banner → отображается в S03)
   - Приоритет в каталоге (M03 → priority_boost → отображается в S04)
   - Пуш-уведомление (M03 → push_notification)
4. Отслеживает метрики до/после (M03 → metrics_before_after):
   - Рост просмотров
   - Рост активаций
5. Конец периода → статус: Промо → Активна (M05)

**Exit criteria:** benefit-энгейджмент продвинут, метрики выросли.

---

### J26 — Закрытие benefit-энгейджмента (партнёра)

**Триггер:** провайдер закрыл программу или прекратил сотрудничество.
**Потребность:** деактивировать benefit-энгейджмент, уведомить сотрудников.
**Артефакты:** M01, M05, M02, B01

**Путь:**
1. Менеджер деактивирует offer (M05 → archived) → `PATCH /admin/engagements/:id/status` → скрыт из каталога (S04)
2. Уже подключенные сотрудники (UserEngagement instances):
   - Уведомление «Ваш benefit X отключен» (S03)
   - Предложение альтернативы
3. Если есть активный период → status → Завершена (M05)
4. Возврат баллов за остаток периода (B01 → credit)
5. Менеджер ищет замену → добавляет нового провайдера как benefit type (J21)

**Exit criteria:** benefit-энгейджмент деактивирован, сотрудники уведомлены, замена найдена.

### J27 — Сезонный набор benefit-энгейджментов

**Триггер:** наступил новый сезон / корпоративный праздник / годовщина.
**Потребность:** создать тематический набор benefit-энгейджментов.
**Артефакты:** M04, M03, M02

**Путь:**
1. Менеджер формирует набор (M04, `POST /admin/collections`):
   - Название
   - Состав из offers (только benefit)
   - Период действия
2. Настраивает баннер набора (M04 → banner):
   - Отображается на Главной (S03)
   - Отображается в Каталоге (S04)
3. Публикует набор → сотрудники видят (J13a)
4. Отслеживает метрики (M04 → metrics):
   - Конверсия набора
   - GMV набора
5. Конец периода → набор архивирован

**Exit criteria:** набор benefit-энгейджментов активен, сотрудники реагируют, метрики собираются.

### J28 — Управление контентом

**Триггер:** нужно обновить FAQ, баннеры, описания льгот.
**Потребность:** поддерживать актуальный контент.
**Артефакты:** M07

**Путь:**
1. Менеджер редактирует FAQ (M07 → faq):
   - Добавляет/удаляет вопросы
   - Обновляет ответы
   - → отображение в ЛК-1 (S07)
2. Обновляет баннеры на Главной (M07 → banners):
   - Тексты, изображения, ссылки
   - → отображение в ЛК-1 (S03)
3. Редактирует описания карточек (M07 → card_descriptions):
   - Названия, описания, условия
   - → отображение в ЛК-1 (S04)
4. Сохраняет → изменения публикуются мгновенно

**Exit criteria:** контент актуален, опубликован.

### J29 — Настройка рекомендательных правил

**Триггер:** нужно изменить логику рекомендаций (новый сегмент, новый контекст).
**Потребность:** добавить или изменить правило рекомендаций.
**Артефакты:** M06

**Путь:**
1. Менеджер открывает «Рекомендации» (M06)
2. **Контекст** (слой 1):
   - Изменяет порог «дни_до_сгорания < N»
   - Добавляет новый контекст (пример: день зарплаты)
3. **Сегмент** (слой 2):
   - Создаёт новый сегмент: название, критерии, список карточек (M01)
   - Устанавливает приоритет при конфликте
4. Тестирует правило: «какое правило сработало для сотрудника X» (M06 → отладка)
5. Публикует → правила применяются в реальном времени (S03, S04)

**Exit criteria:** правила обновлены, применяются корректно.

### J30 — Жизненный цикл benefit-энгейджмента: полный путь

**Триггер:** benefit-энгейджмент проходит все стадии жизни.
**Потребность:** проконтролировать полный цикл от создания до закрытия.
**Артефакты:** M01, M05, M02, B01

**Путь:**
1. Создание engagement type (type=benefit) → Черновик (J21)
2. Создание engagement offer → Публикация → available (J21)
3. Продвижение → `PATCH /admin/engagement-types/:id/catalog-status?=seasonal` (J25)
4. Конец промо → available (J25)
5. Временное скрытие → `PATCH /admin/engagements/:id/status?=paused` (J26)
6. Восстановление → active (J26)
7. Финальное закрытие → archived (J26)
8. Метрики на каждом этапе (M02 → `/admin/engagement-types/analytics`)

**Exit criteria:** benefit-энгейджмент прошел полный цикл, метрики зафиксированы.

### J36 — Self-service запрос на подключение нового провайдера

**Триггер:** сотрудник или HR предлагает нового провайдера для каталога льгот.
**Потребность:** форма предложения нового партнёра → очередь рассмотрения → J21.
**Артефакты:** M08 (Provision Request Queue), S07, M01

**Путь:**
1. Сотрудник заполняет форму «Предложить провайдера» из раздела Поддержка (S07) или отдельная ссылка в Каталоге (S04)
2. Данные: название провайдера, категория льгот, обоснование, ссылка на сайт
3. Заявка попадает в очередь M08 → менеджер каталога видит список предложений с метрикой «популярность» (сколько раз предлагали)
4. Менеджер оценивает → если OK → переходит к J21 (полное создание engagement type)
5. Если отказ → сотрудник получает уведомление с причиной (S03 / email)
6. HR также может подать заявку из H01 «Предложить провайдера для отдела»

**Exit criteria:** предложение подано, оценено менеджером, провайдер добавлен в каталог (или отказ мотивирован).

### J42 — A/B тестирование описаний и баннеров

**Триггер:** менеджер хочет повысить конверсию промо-кампаний.
**Потребность:** A/B тесты карточек льгот, баннеров, описаний.
**Артефакты:** M09 (A/B Testing Engine), M02, M03, M07

**Путь:**
1. Менеджер открывает M09 → выбирает engagement offer или баннер (M07)
2. Создаёт варианты A и B:
   - Для карточки: разное название, описание, иконка, изображение
   - Для баннера: разный текст, изображение, CTA-текст
3. Настраивает расщепление трафика: 50/50 или кастомное (30/70)
4. Запускает тест → период N дней
5. M09 собирает метрики: conversion_rate_A, conversion_rate_B, significance, winner
6. После окончания → автоматически применяется победитель (или менеджер подтверждает вручную)
7. Результаты логируются в M02 analytics

**Exit criteria:** тест запущен, данные собраны, победитель определён, winner применён.

### J44 — White-label: настройка email-шаблонов tenant'ом

**Триггер:** каждый tenant хочет свои branded email-уведомления вместо заглушек.
**Потребность:** визуальный редактор шаблонов email/SMS.
**Артефакты:** M07 (расширение), S03, H06

**Путь:**
1. Менеджер открывает M07 → «Шаблоны уведомлений»
2. Видит список типов: регистрация, начисление, сгорание, апрув, массовые рассылки (H06)
3. Редактирует шаблон в WYSIWYG-редакторе:
   - Переменные: {user_name}, {amount}, {product_name}, {deadline}
   - Brand-цвета (из tenant brand_css), логотип
   - HTML-шаблон для email, текстовый для SMS
4. Предпросмотр → «Отправить тестовый» на свой email
5. Публикует → шаблон применяется ко всем уведомлениям tenant'а
6. Версионирование: rollback к предыдущей версии одним кликом

**Exit criteria:** шаблоны настраиваются без кода, применяются в реальном времени, имеют версии и превью.
