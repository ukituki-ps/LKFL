**Артефакты:** S01–S10, B01, M02, M06, H05, H07

### J01 — Новый сотрудник: знакомство с системой

**Триггер:** сотрудник вышел на новую работу, получил корпоративную почту.
**Потребность:** понять, какие льготы доступны, зарегистрироваться, начать пользоваться.
**Артефакты:** S01, S02, S03

**Путь:**
1. HR-система синхронизирует сотрудника через REST API (H01)
2. Система отправляет email-уведомление со ссылкой на регистрацию (S01)
3. Сотрудник переходит по ссылке → открывает страницу регистрации (S01)
4. Заполняет форму: ФИО, email, дата рождения, телефон, пароль, 2 галки согласий (S01)
5. Система проверяет пару email+ДР по реестру → успех (S01)
6. Сотрудник входит в систему → видит Главную страницу (S02 → S03)
7. Видит баланс, активные льготы, быстрые действия, рекомендации (S03)

**Exit criteria:** сотрудник зарегистрирован, вошёл в систему, видит Главную страницу.

### J02 — Период распределения открыт: выбор программы льгот (type=benefit engagements)

**Триггер:** HR открыл период распределения. Сотрудник получил уведомление.
**Потребность:** распределить баллы между категориями benefit-энгейджментов по своему усмотрению.
**Артефакты:** S03, S04, S05, S09, S10

**Путь:**
1. Сотрудник входит → видит уведомление об открытом периоде (S03)
2. Переходит в Каталог энгейджментов → фильтр `type=benefit` → изучает доступные офферы (S04)
3. Открывает детальную модалку → изучает условия (S10)
4. Для ДМС: запускает Engagement Wizard → выбирает опцию → выбирает оплату → подтверждает (S09)
5. Система обновляет баланс → отражает списание (billing_direction=debit) (S05)
6. При необходимости подключает родственников (J04, J14)
7. Остаток баллов остаётся на счёте / конвертируется по правилу (B01)

**Exit criteria:** сотрудник сформировал программу benefit-энгейджментов, баланс обновлён, льготы активированы.

### J03 — Нужна медицинская помощь: использование ДМС

**Триггер:** сотрудник заболел, нужна медицинская помощь.
**Потребность:** быстро найти клинику, показать полис, понять, что покрывается.
**Артефакты:** S10, S06

**Путь:**
1. Сотрудник входит → открывает модалку «ДМС — Базовая программа» (S10)
2. Таб «Условия программы» → изучает покрытия (S10)
3. Таб «Мой полис» → видит номер полиса, срок действия, копирует/скачивает (S10 → S06)
4. Таб «Клиники на карте» → ищет клинику, видит адрес и график (S10)
5. Идёт в клинику → называет номер полиса в регистратуре
6. Параллельно может позвонить на горячую линию страховой

**Exit criteria:** сотрудник получил медицинскую помощь по полису ДМС.

### J04 — Хочу расширить страховку на семью

**Триггер:** сотрудник хочет подключить супруга/детей/родителей к ДМС.
**Потребность:** добавить застрахованных лиц к существующей программе.
**Артефакты:** S03, S04, S10, R01

**Путь:**
1. Сотрудник нажимает «Добавить родственника к ДМС» (S03 → быстрые действия или S04 → ДМС Семья)
2. Открывается модалка → сотрудник заполняет анкету: ФИО, родство, email, ДР, телефон, пол, адрес, программа ДМС (S10)
3. Для родственников старше 18 лет: система отправляет SMS со ссылкой на согласие (J14a)
4. Для несовершеннолетних: сотрудник подтверждает согласие за ребёнка (J14b)
5. Система валидирует → добавляет в реестр прикрепления ДМС
6. Статус на Главной: «Ожидает» → «Активна» после подтверждения страховщика (S03)

**Exit criteria:** родственник добавлен в полис ДМС, статус активен.

### J05 — Апгрейд ДМС: улучшение программы

**Триггер:** сотрудник хочет расширить покрытие (стоматология, офтальмология, VIP-клиники).
**Потребность:** перейти с базовой программы на расширенную или премиум.
**Артефакты:** S03, S04, S09, S10, S06

**Путь:**
1. Сотрудник нажимает «Апгрейд ДМС» (S03 → быстрые действия или S04 → карточка)
2. Открывается ДМС Wizard (S09):
   - Шаг 1: выбор программы (карточки с ценой)
   - Шаг 2: выбор оплаты (удержание из ЗП / оплата картой — S08)
   - Шаг 3: текст заявления + чекбокс согласия на ПДн
   - Шаг 4: «Заявка принята» → данные переданы страховщику
3. Статус на Главной: «Ожидает» → «Активна» после подтверждения (S03)
4. Система формирует документ → появляется в разделе «Документы» (S06)

**Exit criteria:** заявка на апгрейд подана, документ сформирован, статус обновлён.

### J06 — Появился ребёнок: корпоративный маткапитал

**Триггер:** у сотрудника родился ребёнок.
**Потребность:** получить единовременную выплату от компании.
**Артефакты:** S03, S09, S10, S06

**Путь:**
1. Сотрудник нажимает «Заявка на маткапитал» (S03 → быстрые действия)
2. Открывается Маткапитал Wizard (S09):
   - Шаг 1: условия (размер выплаты, стаж, срок заявки)
   - Шаг 2: данные ребёнка (ФИО, ДР, номер свидетельства, дата выдачи)
   - Шаг 3: текст заявления + чекбокс согласия
   - Шаг 4: «Заявка отправлена» → HR свяжется
3. HR проверяет документы → одобряет (H04)
4. Выплата в течение 10 рабочих дней после одобрения
5. Статус заявки в разделе «Документы» (S06)

**Exit criteria:** заявка подана, HR одобрил, выплата произведена.

### J07 — Хочу развиваться: обучение и wellbeing (type=benefit engagement)

**Триггер:** сотрудник хочет прокачать навыки или позаботиться о ментальном здоровье.
**Потребность:** подключиться к benefit-энгейджменту (debit).
**Артефакты:** S04, S10, S05, S06

**Путь:**
1. Сотрудник открывает Каталог энгейджментов → фильтр `type=benefit` по категории (S04)
2. Изучает карточки — оффер, стоимость, описание (S04)
3. Открывает модалку → видит детали (S10)
4. Нажимает «Подключить» → POST /user-engagements → billing_direction=debit → баланс обновлён (S05)
5. Для benefit-энгейджментов, требующих апрува: статус «Ожидает» → HR подтверждает (H04) → «Активна»
6. Система формирует документ → появляется в «Документы» (S06)

**Exit criteria:** benefit-энгейджмент подключен, баллы списаны, доступ к сервису провайдера получен.

### J08 — Не хватает баллов: управление балансом

**Триггер:** сотрудник хочет льготу, но баланс недостаточен.
**Потребность:** понять текущий баланс и историю.
**Артефакты:** S05

**Путь:**
1. Сотрудник открывает «Мои баллы» (S05)
2. Видит текущий баланс по категориям, дату сгорания, историю транзакций (S05)
3. Фильтрует транзакции: все / начисления / списания (S05)
4. Видит, каких категорий не хватает → решает: доплатить (J08a) / подождать начисления (T01) / выбрать другую льготу

**Exit criteria:** сотрудник видит баланс и историю, понимает нехватку.

### J08a — Доплата при нехватке баланса

**Триггер:** сотрудник подключил льготу, но баланс недостаточен (J08).
**Потребность:** оплатить разницу.
**Артефакты:** S08, B01

**Путь:**
1. Система показывает нехватку: «Не хватает N баллов» (S08)
2. **Вариант А — оплата картой** (S08):
   - Форма карты → ввод данных → заморозка
   - Подтверждение → транзакция создана (B01)
3. **Вариант Б — удержание из ЗП** (S08):
   - Текст заявления → чекбокс согласия → отправка на апрув HR (H04)
   - HR одобряет → удержание при начислении ЗП
4. Льгота активируется → баланс обновлён (S05)

**Exit criteria:** льгота подключена, нехватка оплачена, транзакция подтверждена.

### J09 — Возник вопрос: поддержка

**Триггер:** у сотрудника возникла проблема или вопрос.
**Потребность:** получить ответ быстро.
**Артефакты:** S07

**Путь:**
1. Сотрудник открывает «Поддержка» → изучает FAQ (S07)
2. Нашёл ответ → проблема решена
3. Не нашёл → заполняет форму обращения (тема + сообщение) → отправляет (S07)
4. Получает ответ на email в течение N рабочих дней

**Exit criteria:** вопрос решён через FAQ или обращение в поддержку.

**Триггер:** сотруднику нужны документы для подтверждения льготы, полиса или согласия.

**Артефакты:** S06, S10

**Путь:**
1. Сотрудник открывает «Документы» (S06)
2. Видит таблицу: заявление, согласие, полис ДМС (S06)

### J11 — Партнёрские сервисы: сквозная авторизация

**Триггер:** сотрудник хочет воспользоваться сторонним сервисом по льготной цене.

**Путь:**

2. Открывается новая вкладка → авторизация происходит автоматически (E01)
3. Сотрудник использует сервис провайдера

**Exit criteria:** сотрудник авторизован на стороннем сервисе без повторного входа.

### J12 — Выполнение activity: заработок бонусных баллов (type=activity engagement, credit)

**Триггер:** система отправила уведомление о доступном engagement-е типа activity.
**Потребность:** выполнить действие → получить бонусные баллы (credit).
**Артефакты:** S03, S09, S05

**Путь:**
1. Сотрудник получает уведомление (S03 → event_feed / push / email)
2. Переходит к каталогу → фильтр `type=activity` → видит список доступных (S04)
3. **Тип «опрос»** (S09 Engagement Wizard):
   - POST /user-engagements (начало) → отвечает на N вопросов → POST /user-engagements/:id/completes → начисление баллов
4. **Тип «событие»** (S09):
   - Подтверждает участие → POST /user-engagements/:id/completes → начисление баллов
5. Система начисляет бонусные баллы → транзакция credit создана (B01, billing_direction=credit)
6. Лента событий: «Начислено N баллов» (S03)
7. Баланс обновлён (S05)

**Exit criteria:** activity-энгейджмент выполнен, бонусные баллы начислены.

### J13 — Сгорание баллов: срочное использование

**Триггер:** система отправила напоминание за N дней до конца периода.
**Потребность:** успеть потратить баллы до сгорания.
**Артефакты:** S05, S03, S04, B01

**Путь:**

2. Открывает «Мои баллы» → видит остаток, дату сгорания, предупреждение «Пора потратить» (S05)
3. Переходит в Каталог → выбирает льготы на весь остаток (S04)
4. Подключает нужные → баланс обновлён (B01)

**Exit criteria:** баллы потрачены до конца периода.

### J13a — Сезонный набор льгот

**Триггер:** наступил сезон / корпоративный праздник / годовщина. Менеджер создал набор (J27).
**Потребность:** быстро подключить тематический набор льгот одной кнопкой.
**Артефакты:** S03, S04, S05, M04

**Путь:**
1. Сотрудник видит баннер набора на Главной (S03) или в Каталоге (S04)
2. Открывает набор → видит состав льгот и общую стоимость (M04)
3. Нажимает «Подключить набор» → система проверяет баланс (B01)
4. При нехватке: предложение доплаты (J08a)
5. Все льготы набора подключены → баланс обновлён (S05)
6. Статус набора на Главной: «Подключено» (S03)

**Exit criteria:** набор подключён, баллы списаны, все льготы активированы.

### J13b — Рекомендации: персональные предложения

**Триггер:** сотрудник заходит в систему. Система подбирает релевантные предложения.
**Потребность:** видеть предложения, подходящие именно мне.
**Артефакты:** S03, S04, M06

**Путь:**
1. Сотрудник входит → система вычисляет сегмент + контекст (M06)
2. **Контекст:** период открыт → баннер; баланс горит → «Пора потратить» (M06 слой 1)
3. **Сегмент:** junior/staff → фитнес+мерч; parent → семья (M06 слой 2)
4. Рекомендации отображаются на Главной (S03) и в Каталоге (S04)
5. Сотрудник нажимает на рекомендацию → переходит к карточке / подключает

**Exit criteria:** сотрудник видит персональные предложения, релевантные его сегменту и контексту.

### J13c — Удаление персональных данных

**Триггер:** сотрудник запросил запрет обработки ПДн (письменно или через систему).
**Потребность:** удалить данные, сохранить доступ к неотзываемым льготам.
**Артефакты:** S07, S02

**Путь:**
1. Сотрудник обращается в поддержку (S07) → тема: «Удаление ПДн»
2. Система:
   - Снимает все согласия
   - Каскадная деактивация льгот, требующих ПДн
   - Блокировка доступа к ЛК (S02)
   - Удаление персональных данных из БД
   - Уведомление провайдеров о прекращении обработки
3. HR получает уведомление (H01)

**Exit criteria:** данные удалены, согласия сняты, сотрудник уведомлён.

### J14a — Родственник старше 18 лет: согласие через SMS

**Триггер:** сотрудник ввёл данные родственника (J04, шаг 2).
**Потребность:** дать согласие на обработку ПДн.

### J14b — Родственник несовершеннолетний: согласие родителя

### J31 — Сотрудник переключает между продуктами tenant'а

**Триггер:** у tenant'а несколько продуктов с разными программами льгот; сотрудник работает в нескольких департаментах.
**Потребность:** переключиться между продуктами и видеть соответствующий каталог, баланс, льготы.
**Артефакты:** S03, S04, S05

**Путь:**
1. Сотрудник видит переключатель продуктов в хедере (S03) — список продуктов tenant'а, к которым у него доступ
2. Выбирает продукт → интерфейс пересобирается: каталог (S04), баланс (S05), активные льготы
3. Баланс продукта A и продукта B изолированы — видит раздельные счета
4. Рекомендации (M06) адаптируются к выбранному продукту
5. История переключений логгируется (H07)

**Exit criteria:** сотрудник видит правильный каталог и баланс для выбранного продукта, данные изолированы по продуктам.

### J34 — Referral program: сотрудник приводит коллегу

**Триггер:** HR создал referral-энгейджмент как type=activity (J18).
**Потребность:** привести нового коллегу → получить бонус (двойное вознаграждение).
**Артефакты:** S03, S04, S09, S05, B01

**Путь:**
1. Сотрудник видит карточку «Приведи коллегу» в каталоге `type=activity` (S04)
2. Открывает модалку → получает уникальную реферальную ссылку + QR-код (S10)
3. Делится ссылкой с кандидатом
4. Кандидат регистрируется через ссылку → система связывает приводящего сотрудника (S01)
5. Кандидат проходит испытательный срок N месяцев → система начисляет баллы приводящему (B01, credit)
6. Статус referral'а на Главной: «Активен» → «Успешно» / «Не успешно» (S03)
7. Метрики для HR: сколько привёл, конверсия «ссылка → регистрация → испытательный» (H05)

**Exit criteria:** referral-ссылка создана, коллега зарегистрирован, бонус начислен при успешном прохождении испытательного срока.

### J35 — Командные цели и челленджи

**Триггер:** HR создал командный челлендж как type=activity (J18, engagement_type=team_challenge).
**Потребность:** выполнить цель командно (отдел/департамент) → получить коллективный бонус.
**Артефакты:** S03, S04, S09, S05, B01

**Путь:**
1. Сотрудник видит командный челлендж на Главной (S03 → event_feed) и в каталоге (S04)
2. Карточка показывает: цель отдела (например, 80% участия), текущий прогресс-бар, список участников
3. Сотрудник подтверждает участие → счётчик отдела увеличивается (S09)
4. При достижении порога X%: всем участникам отдела начисляется бонус (B01, credit)
5. Лента: «Отдел Marketing достиг цели! +500 баллов каждому» (S03)
6. HR видит метрики: participation% по отделам, winner department, overall bonus_spent (H05)

**Exit criteria:** командная цель достигнута, бонус начислен всем участникам отдела.

### J41 — Survey/NPS после подключения льготы

**Триггер:** сотрудник завершил activation flow benefit-энгейджмента (J02, J05, J07).
**Потребность:** оценить опыт → система собирает фидбек.
**Артефакты:** S03, S09, S05, M02

**Путь:**

2. Ответ сохраняется → транзакция: «Survey выполнен» (+ бонус N баллов как credit, B01)

### J43 — Перевод баланса между продуктами tenant'а

**Триггер:** у сотрудника есть неиспользованные баллы в продукте А, но нужны в продукте Б.
**Потребность:** перенести остаток между продуктами.
**Артефакты:** S05, B01

**Путь:**
1. Сотрудник открывает «Мои баллы» → видит баланс по каждому продукту (S05 — группировка по product_id)
2. Нажимает «Перевести» → выбирает источник продукта и целевой продукт
3. Вводит сумму → система валидирует:
   - Достаточно ли в источнике
   - Нет ли active debit-транзакций, которые могут заблокировать перевод
4. Подтверждает → Billing Rule делает debit с источника, credit на цель (B01)
5. История перевода видна в транзакциях обоих продуктов (S05)

**Exit criteria:** баллы перенесены между продуктами, транзакции зафиксированы в обоих счётах.
