# T0101 — Формирование раздела «Контекст»

## Веха

M01-создание-описания

## Раздел

`doc/контекст/`

## Контекст

Раздел «Контекст» — верхний уровень документации. На его основе строятся Архитектура и Vision. Сейчас в `doc/контекст/` только README.md с описанием структуры. Нужно создать полноценные файлы раздела на основе артефактов заказчика.

## Источник данных

Все артефакты в `doc/артефакты/`:
- **Функциональные требования v1** — разделы 1-4 (назначение, глоссарий, основные функции)
- **Прототип ЛК** — страницы, модалки, данные (для понимания акторов)
- **Процесс регистрации** — роли участников процесса
- **Кубики** — бизнес-контекст, варианты реализации, сроки

## Что сделать

Создать 4 файла в `doc/контекст/`:

### 1. `проблема.md`

Описать бизнес-проблему:
- Почему нужна платформа гибких льгот
- Какие боли решает (для сотрудников и HR)
- Цели проекта (из ФТ §2 «Назначение»)

### 2. `акторы.md`

Описать роли пользователей:
- Сотрудник (физлицо)
- Администратор платформы
- HR администратор
- Система Заказчика
- FTP Сервер
- Внешние провайдеры (ДМС, фитнес, питание, мерч)

Для каждого актора:
- Описание роли
- Что делает в системе
- Взаимодействие с другими акторами

### 3. `ограничения.md`

Описать ограничения:
- **Масштаб:** 100 000+ активных пользователей, горизонтальное масштабирование
- **Безопасность:** OWASP, шифрование, валидация паролей, сканирование уязвимостей
- **Отказоустойчивость:** резервирование БД, балансировка, sticky sessions, кластер
- **Браузеры:** IE, Firefox, Opera, Safari, Chrome; Mobile: iOS Safari, Android Chrome
- **Комплаенс:** ФСТЭК, 152-ФЗ, единое место хранения данных
- **Сроки:** запуск лето 2026, полная работа август 2026

### 4. `negative-criteria.md`

Описать что НЕ делаем:
- Не строим monolith с монолитным фронтом (SPA уже есть)
- Не храним полисы и документы на платформе (только в ядре)
- Не обрабатываем ПДн без галки согласия
- Не создаём задачу без brief.md → plan.yaml → report.md
- Не меняем систему номенклатуры путей/файлов

## Критерии приёмки

- [ ] `проблема.md` — описана бизнес-проблема и цели
- [ ] `акторы.md` — описаны все 6+ ролей с взаимодействиями
- [ ] `ограничения.md` — описаны все нефункциональные требования
- [ ] `negative-criteria.md` — описаны negative criteria
- [ ] Все файлы согласованы с артефактами заказчика
