# M03 — Архитектура льгот

## Цель

Пересобрать архитектуру льгот как систему из 5 сущностей вместо одной монолитной схемы.
Обеспечить покрытие **абсолютно любых льгот**: подписки, пер-юз, гибридные потоки, тарифы, наборы.

## Описание

T0203 добавил YAML-схему benefit в `модули.md`. Анализ показал, что схема покрывает
только одну категорию льгот (подписка с фиксированной ценой). Шесть критических проблем
выявлены в ходе code review документации.

Решение: разложить «льготу» на 5 сущностей:

```
┌──────────────┐     ┌──────────────┐     ┌───────────────────┐
│  Benefit     │ 1 ──┤  BenefitPlan │ 1 ──┤ ActivationFlow    │
│ (каталог)    │     │ (тариф)      │     │ (поток)           │
└──────────────┘     └──────┬───────┘     └───────────────────┘
                            │
                    ┌───────┴───────┐
                    │               │
            ┌───────┴───────┐  ┌────┴──────────────┐
            │ Eligibility    │  │ Billing Rule      │
            │ (доступ)       │  │ (списание)         │
            └────────────────┘  └───────────────────┘
```
