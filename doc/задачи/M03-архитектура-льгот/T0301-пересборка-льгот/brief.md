# T0301 — Пересборка раздела «Льготы» с нуля

## Веха

M03-архитектура-льгот

## Контекст

T0203 добавил YAML-схему benefit-сущности в `архитектура/модули.md`. При проверке схемы выяснилось, что она **не покрывает «абсолютно любые льготы»** — ключевое требование платформы.

Схема была написана под конкретный пример (фитнес-абонемент) и отражает ментальную модель одной категории льгот. Шесть выявленных проблем:

| # | Проблема | Критичность |
|---|----------|-------------|
| 1 | `cost.currency` — жёсткий enum из 3 вариантов | Высокая — блокирует tenant с другой валютой |
| 2 | `cost.frequency` — нет `per_use` / `per_transaction` | Высокая — не описывается ДМС, стоматология, подарочные карты |
| 3 | `activation.type` — один enum, не покрывает цепочки шагов | Высокая — ДМС (approval → form → redirect) не описывается |
| 4 | `eligibility` — плоский массив, нет AND/OR логики | Средняя — реальные бизнес-правила не выражаются |
| 5 | Нет поддержки наборов льгот (bundles) | Средняя — J13a, J27 требуют коллекций |
| 6 | Нет уровней/вариантов (tiers) одной льготы | Средняя — J05 (апгрейд ДМС) требует plans |

### Анализ существующей архитектуры показал

Проанализированы journeys (J01–J30), Billing Rule engine, Activity Type model, Integration Adapter interface, матрица настраиваемости, tenant-onboarding.

**Вывод:** «льгота» — не одна сущность. Одна YAML-схема пытается описать одновременно: каталог, тарифы, поток активации, доступ, биллинг, коллекции. Для произвольных льгот это невозможно.

### Почему одна схема не работает

- ДМС: один продукт → три тарифа (базовый/расширенный/премиум) → активация через approval + form + redirect → списание per_use
- Фитнес: один продукт → подписка monthly → активация instant → списание monthly
- Маткапитал: один продукт → единовременная выплата → активация form + approval → списание one_time
- Подарочная карта: один продукт → персонифицированная → активация provider_redirect → списание per_transaction

Каждый из этих потоков **не вписывается** в одну схему с фиксированными enum-полями.

## Что сделать

Пересобрать архитектуру льгот как **пять сущностей** вместо одной:

1. **Benefit** — продукт в каталоге (название, описание, провайдер, категория)
2. **BenefitPlan** — тариф/вариант продукта (стоимость, условия, activation_flow, eligibility)
3. **ActivationFlow** — цепочка шагов подключения (массив шагов с порядком)
4. **BenefitCollection** — набор льгот (сезонные, тематические)
5. **UserBenefit** — экземпляр льготы конкретного сотрудника (статус, план, дата активации)

Связь: `Benefit 1 → N BenefitPlan → 1 ActivationFlow + 1 Eligibility`
