# T0203 — Data schema для benefit-сущности

## Веха

M02-улучшение-документации

## Контекст

В документации есть YAML-schema для Billing Rule (`архитектура/биллинг-движок.md`)
и Activity Type (`архитектура/активности.md`). Для benefit-сущности — только текстовое описание:
> «Льгота = карточка с названием, стоимостью, категорией, провайдером и правилами подключения»

Это создаёт дисбаланс: биллинг и активности выглядят детально спроектированными,
а льготы — «картинка + текст». Для разработчика это сигнал: льготы — less important.

## Что сделать

Обеспечить детализацию benefit-сущности на уровне Billing Rule и Activity Type:
YAML-схемы, примеры, диаграммы связей.

### Почему отдельный файл

Изначальная задача предполагала вставку одной YAML-схемы в `модули.md`.
Анализ показал, что «льгота» — не одна сущность, а система из пяти (T0301):
Benefit, BenefitPlan, ActivationFlow, BenefitCollection, UserBenefit.

Полная документация → [`архитектура/льготы.md`](../../архитектура/льготы.md) (~800 строк).
В `модули.md` — краткая ссылка с описанием.

## Критерии приёмки

- [x] Benefit-сущность документирована на уровне Billing Rule / Activity Type (YAML-схемы + примеры)
- [x] Формат консистентен с Billing Rule (`биллинг-движок.md`) и Activity Type (`активности.md`)
- [x] Комментарии к полям на русском
- [x] `модули.md` содержит ссылку на `льготы.md`
- [x] `архитектура/льготы.md` существует и содержит полные схемы
- [x] Схемы покрывают все типы льгот (подписка, per_use, one_time, per_session)
