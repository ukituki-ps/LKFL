## Краткое описание

Платформа взаимодействует с внешними системами льгот через **Integration Proxy** (`lkfl-integration-proxy`, :8090) — отдельный бинарник, изолирующий все исходящие HTTP-вызовы к провайдерам (ADR-035, M16). Монолит (`lkfl-server`, :8080) общается с proxy через gRPC на localhost. Прямые вызовы от монолита к внешним провайдерам запрещены (за исключением SSO redirect через Keycloak).

> **Принцип:** количество провайдеров не фиксировано — «сколько есть, столько и будет». Каждый провайдер — конфигурационный объект (name, protocol, endpoints, auth_method).
> **M07 T0704:** 9 из 11 адаптеров управляются через YAML config (Generic REST adapter, ADR-017). Новый REST-провайдер = YAML файл, не Go-код. SAML/проприетарные протоколы остаются hard-coded.
> **M12:** межмодульные вызовы были direct Go call через `integrations.ProviderGateway.*()`.
> **M16 ADR-035:** ProviderAdapter реализации перенесены в proxy (`integration-proxy/adapters/`). Монолит использует gRPC client (`internal/integrationclient/`) для вызовов proxy.

## Архитектура интеграций

```
                     ┌─────────────────────────────┐
                     │         Nginx (:80)         │
                     │                             │
                     │  /api/v1/*     → :8080      │
                     │  /webhooks/*   → :8091      │
                     └──────┬──────────────┬───────┘
                            │              │
                ┌───────────▼──┐    ┌──────▼──────────────┐
                │  lkfl-       │    │ lkfl-integration-   │
                │  server      │    │ proxy               │
                │  (:8080)     │    │ (:8090 gRPC)        │
                │              │    │ (:8091 HTTP)        │
                │  gRPC client │───→│  Provider adapters  │
                │  (localhost) │    │  Circuit breaker    │
                └──────────────┘    │  Retry/timeout      │
                                    │  Webhook receiver   │
                                    │  Credential store   │
                                    │  Worker pool        │
                                    └────────┬────────────┘
                                             │
                                    ┌────────▼────────┐
                                    │  Внешние         │
                                    │  провайдеры      │
                                    │  (HTTP/REST)     │
                                    └─────────────────┘
```

**Принцип (M16):** Монолит не делает прямых HTTP-вызовов к внешним провайдерам. Все исходящие вызовы идут через gRPC к proxy, который выполняет HTTP call к провайдеру в изолированном процессе.

### Принцип «Единый шлюз + Кэш»

1. **Единый шлюз — Integration Proxy:** Монолит никогда не обращается к внешним провайдерам напрямую. Все исходящие вызовы к льготным провайдерам идут через `lkfl-integration-proxy` (gRPC на localhost). Proxy реализует ProviderAdapter для каждого провайдера.
2. **SSO-редирект:** Keycloak Identity Broker перенаправляет пользователя на внешний сервис (OIDC/SAML redirect). Платформа не участвует в token exchange.
3. **Локальный кэш:** Proxy хранит кэшированную копию каталогов и статусов в PostgreSQL (schema `lkfl_integration`). Обновляется по расписанию (worker proxy) или по webhook. Монолит читает кэш через gRPC `GetCatalog()` / `GetProviderStatus()`.
4. **Отказоустойчивость:** Если провайдер недоступен → circuit breaker proxy → кэш используется как fallback. Монолит получает статус из кэша proxy. Пользователь видит «обновляется».
5. **Асинхронная активация:** Операции `Activate`/`Deactivate` выполняются асинхронно в worker pool proxy. Монолит получает `job_id` за < 10ms и уведомляется о результате через webhook callback.

## ProviderAdapter interface

> **M16 ADR-035:** Интерфейс определён в `proto/integration/v1/integration.proto` (gRPC contract). Реализация каждого адаптера находится в `integration-proxy/adapters/` (proxy бинарник), не в монолите. Монолит взаимодействует с proxy через gRPC client (`internal/integrationclient/`), не вызывая адаптеры напрямую.

Все адаптеры реализуют единый интерфейс (определён в Go для использования внутри proxy):

```go
type ProviderAdapter interface {
    // Name — уникальный идентификатор провайдера
    Name() string

    // Activate — активировать льготу для пользователя
    Activate(ctx context.Context, req *ActivationRequest) (*ActivationResult, error)

    // Deactivate — отключить льготу
    Deactivate(ctx context.Context, req *DeactivationRequest) error

    // Status — текущий статус льготы у пользователя
    Status(ctx context.Context, userId string) (*ProviderStatus, error)

    // SyncCatalog — синхронизировать каталог энгейджментов от провайдера
    SyncCatalog(ctx context.Context) ([]EngagementOffer, error)

    // Health — проверка доступности провайдера
    Health(ctx context.Context) error

    // Configure — загрузка конфигурации из Integration Proxy (endpoints, credentials, timeout, retry)
    Configure(cfg *ProviderConfig) error
}
```

### Service IntegrationService

```protobuf
service IntegrationService {
  // === Сynchronous — быстрые/idempotent операции (< 5s) ===

  // Получить статус провайдера для пользователя
  // Читает локальный кэш proxy, не вызывает внешний API
  rpc GetProviderStatus(ProviderStatusRequest) returns (ProviderStatusResponse);

  // Получить каталог льгот провайдера
  // Читает кэш из БД proxy (lkfl_integration), не вызывает провайдера
  rpc GetCatalog(CatalogRequest) returns (CatalogResponse);

  // Проверка здоровья proxy и подключённых провайдеров
  rpc HealthCheck(HealthCheckRequest) returns (HealthCheckResponse);

  // === Asynchronous — медленные операции (activate/deactivate) ===

  // Активировать льготу для пользователя
  // Возвращает job_id немедленно (< 10ms). Выполнение в worker pool proxy.
  // Результат доставляется через webhook callback на монолит.
  rpc Activate(ActivateRequest) returns (ActivateResponse);

  // Деактивировать льготу для пользователя
  // Аналогично Activate — асинхронно, через worker pool.
  rpc Deactivate(DeactivateRequest) returns (DeactivateResponse);

  // === Admin — управление провайдерами ===

  // Получить список всех провайдеров
  rpc ListProviders(ListProvidersRequest) returns (ListProvidersResponse);

  // Получить детали конкретного провайдера
  rpc GetProvider(GetProviderRequest) returns (GetProviderResponse);

  // Обновить конфигурацию провайдера
  rpc UpdateProvider(UpdateProviderRequest) returns (UpdateProviderResponse);

  // Запустить ручную синхронизацию каталога провайдера
  rpc TriggerSync(TriggerSyncRequest) returns (TriggerSyncResponse);

  // Получить логи синхронизации для провайдера
  rpc GetSyncLogs(GetSyncLogsRequest) returns (GetSyncLogsResponse);
}
```

## Generic REST adapter (M07 T0704, ADR-017)

> **M16 ADR-035:** Generic REST adapter реализован в proxy (`integration-proxy/adapters/generic_rest.go`). YAML-конфиги провайдеров загружаются proxy из `provider-configs/` при старте и при hot-reload. Монолит не знает о YAML-конфигурации — он вызывает proxy по gRPC.

9 из 11 адаптеров заменяются на Generic REST adapter — YAML-config + template-based transform. YAML-конфиги хранятся в `provider-configs/` и загружаются proxy.

### YAML-config format

```yaml
# provider-configs/alpha.yaml
name: "alpha"
protocol: "rest"
endpoints:
  activate:
    method: "POST"
    url: "${ALPHA_BASE_URL}/v1/activate"
    headers:
      Authorization: "Bearer ${ALPHA_API_KEY}"
      Content-Type: "application/json"
  deactivate:
    method: "DELETE"
    url: "${ALPHA_BASE_URL}/v1/activate/${userId}"
  status:
    method: "GET"
    url: "${ALPHA_BASE_URL}/v1/status/${userId}"
  sync_catalog:
    method: "GET"
    url: "${ALPHA_BASE_URL}/v1/catalog"
transform:
  request: "alpha/transform_request.tmpl"
  response: "alpha/transform_response.tmpl"
health:
  url: "${ALPHA_BASE_URL}/health"
  interval: "300s"
timeout: "30s"
retry:
  max_attempts: 3
  backoff: "1s,2s,4s"
```

### Как работает

1. **Proxy загружает YAML** — при старте proxy читает `provider-configs/*.yaml` и создаёт адаптеры для каждого REST-провайдера
2. **Call(endpoint, data)** — proxy читает YAML config → выполняет HTTP method + URL к внешнему провайдеру → возвращает raw response
3. **Transform(request, response)** — proxy применяет Go-шаблон (`.tmpl`) для нормализации JSON → `EngagementOffer` / `ActivationResult` → результат доступен монолиту через gRPC `GetCatalog()` / `Activate()`
4. **Health()** — proxy выполняет периодический health check + error rate monitoring → состояние circuit breaker обновляется в `lkfl_integration.circuit_breaker_state`

### Исключения (hard-coded адаптеры обязательны)

| Протокол | Провайдер | Почему hard-coded |
|-----|--------|-----|
| SAML | Ready4, PrimeZone | XML + crypto + assertion validation — не выражается через REST+template |
| Проприетарный | Подарок в квадрате | Бинарный протокол + SLO exchange — требует специфичную библиотеку |
| SOAP | (если появится) | WSDL + WS-Security — требует client generation |

> **Rule of thumb:** если провайдер speaks plain REST + JSON → YAML config. Если SAML/SOAP/проприетарный → hard-coded адаптер.

## Admin endpoints для провайдеров (монолит → proxy через gRPC)

> **M12:** Integrations Hub удалён как отдельный сервис. Admin endpoint'ы были частью `lkfl-server` (internal/api admin router).
> **M16 ADR-035:** Монолит принимает HTTP-запросы от admin UI, но делегирует все операции управления провайдерами к proxy через gRPC. Данные провайдеров (credentials, статусы, конфиги) хранятся в PostgreSQL schema `lkfl_integration` (proxy), не `lkfl_platform`.

**Архитектура admin запросов:**
```
Admin UI → HTTP → lkfl-server (:8080) → gRPC → lkfl-integration-proxy (:8090)
```

Монолит выступает как HTTP facade: принимает запрос от admin UI, вызывает соответствующий gRPC метод proxy, возвращает результат.

**Модули admin UI:**

| Модуль | Назначение | gRPC метод proxy |
|--------|-----------|-----------------|
| Providers | CRUD провайдеров: name, protocol, endpoints, auth_method, status | `ListProviders`, `GetProvider`, `UpdateProvider` |
| Health Dashboard | Статус каждого провайдера: up/down, last_sync, error_rate, latency | `HealthCheck` |
| Sync Control | Manual trigger, schedule management, error log | `TriggerSync`, `GetSyncLogs` |
| Endpoint Config | URL, auth credentials, timeout, retry policy | `UpdateProvider` |
| SLA Monitor | Alert при degrade, escalation rules | `HealthCheck` + метрики |

**API контракты (admin router lkfl-server — HTTP facade → gRPC proxy):**

| Метод | Endpoint | gRPC вызов proxy | Описание |
|-------|----------|-----------------|----------|
| GET | `/admin/providers` | `ListProviders()` | список провайдеров |
| POST | `/admin/providers` | `UpdateProvider()` (создание) | добавление нового провайдера |
| GET | `/admin/providers/:id` | `GetProvider()` | детали + config + health + stats |
| PUT | `/admin/providers/:id` | `UpdateProvider()` | редактирование config |

## Обработка ошибок

> **M16 ADR-035:** Proxy возвращает структурированные ошибки через gRPC. Монолит получает gRPC status + details и преобразует их в HTTP-ответы для фронтенда.

### Ошибки proxy → монолит (gRPC)

Proxy возвращает ошибки через gRPC status codes:

| gRPC Status | Когда | Действие монолита |
|-------------|-------|-------------------|
| `OK` | Успех | Нормальная обработка |
| `UNAVAILABLE` | Proxy недоступен | 503 "сервис интеграций временно недоступен" |
| `DEADLINE_EXCEEDED` | Таймаут gRPC вызова | 504, повтор через retry |
| `INTERNAL` | Ошибка в proxy (circuit breaker open) | 502, использовать кэш |
| `NOT_FOUND` | Провайдер не найден | 404 |
| `INVALID_ARGUMENT` | Невалидный запрос | 400 |

### Ошибки провайдеров (обрабатываются в proxy)

| Сценарий | Стратегия (в proxy) | Результат для монолита |
|----------|-----------|----------------------|
| Timeout провайдера (>30s) | Retry 3x с exponential backoff (1s, 2s, 4s) в worker pool | gRPC `INTERNAL` + circuit breaker increment |
| 3rd retry failure | Dead letter в `lkfl_integration.dead_letters` → alert в Sentry | Job status `FAILED`, webhook callback с ошибкой |
| Circuit breaker | >10 ошибок за 60с → circuit open → cached response | gRPC `INTERNAL`, монолит использует кэш |
| Webhook от провайдера | Proxy: verify signature → validate payload → store in `webhook_events` → HTTP POST `/internal/webhook/callback` → монолит | Монолит обновляет статус, шлёт нотификацию |
| Невалидные данные от провайдера | Reject → log → alert | Dead letter, ручной разбор |
| PCI DSS (пэйшлюз) | Zero card storage → token-only → TLS 1.3 (в `internal/payments/`, не через proxy) | Не затрагивается proxy (пэйшлюз остаётся в монолите) |

## Синхронизация каталога

> **M16 ADR-035:** Синхронизация выполняется worker'ом proxy (не Asynq worker монолита). Каталог хранится в `lkfl_integration` schema (БД proxy). Монолит читает каталог через gRPC `GetCatalog()`.

Каталог льгот хранится локально в `lkfl_integration` БД proxy и обновляется по расписанию:

| Провайдер | Частота | Механизм |
|-----------|---------|----------|
| Все API-провайдеры | Каждые 6 часов | Proxy worker (scheduler) → HTTP call к провайдеру → transform → запись в `lkfl_integration` |
| SSO-провайдеры | По запросу | Manual trigger через gRPC `TriggerSync()` → proxy worker |

> **M11 T1102:** HR-система не участвует в синхронизации через proxy. HR Sync остаётся в монолите (`user/`, Asynq worker `hr-sync-daily`, direct REST call).

Локальный кэш proxy содержит: `benefit_id`, `name`, `description`, `price`, `category`, `provider`, `status`, `updated_at`.

### Flow синхронизации

```
1. Proxy scheduler → пробуждение по расписанию (каждые 6ч)
2. Proxy → HTTP call к внешнему провайдеру (каталог)
3. Proxy → transform (YAML template) → нормализация
4. Proxy → запись в lkfl_integration (каталог таблицы)
5. Запись в provider_sync_log (timestamp, count, errors)
6. Монолит читает актуальный каталог через gRPC GetCatalog() при запросе пользователя
```
