## Детальное описание рисков

### R-001 — Google Fonts (ФСТЭК)

**Категория:** Безопасность | **Критичность:** Высокая | **Статус:** 🟡 Мониторинг

**Описание:** CSP позволяет загрузку шрифтов с `fonts.googleapis.com` / `fonts.gstatic.com` — внешние Google-серверы. ФСТЭК требует, чтобы все зависимости были под контролем оператора (РФ).

**Влияние:**
- Нарушение требований ФСТЭК при сертификации
- Зависимость от внешних серверов (доступность, supply chain)
- Пользовательские данные (UA, IP) отправляются Google

**Митигация:** Self-hosted fonts planned M42. Заменить на локальные шрифты или private CDN в РФ.

**Источник:** Аудит M22 (T2214.1 brief TODO), безопасность.md §9 (CSP)

---

### R-002 — Fail-open rate limiting

**Категория:** Безопасность | **Критичность:** Высокая | **Статус:** 🟢 Принят

**Описание:** Rate limiting middleware (`shared/pkg/middleware/ratelimit.go`) при недоступности Redis переходит в fail-open режим — запросы проходят без ограничений.

**Трейдофф:** Security vs Availability. При падении Redis:
- Fail-closed: все запросы блокируются → DoS для легитимных пользователей
- Fail-open: запросы проходят → риск brute force / DoS

**Влияние:** Если Redis недоступен более ~5 минут, возможны:
- Brute force атаки на auth endpoints
- DoS через catalog endpoints

**Митигация:**
- Nginx-level rate limiting как fallback
- Redis Sentinel / Cluster для HA
- Alerting на Redis downtime

**Источник:** Аудит M22, `shared/pkg/middleware/ratelimit.go`

### R-003 — SRI хэши CSP — заглушка

**Категория:** Безопасность | **Критичность:** Высокая | **Статус:** 🔴 Открыт

**Описание:** CSP header содержит `'sha256-<HASH>'` — заглушку вместо реальных SRI-хэшей inline-скриптов и стилей.

**Влияние:** Если заглушка не заменена до production, CSP не защищает от XSS через inline-скрипты (хотя React auto-escaping снижает риск).

**Митигация:**
1. Собрать production build
2. Вычислить хэши inline ресурсов
3. Подставить в Nginx config
4. CI: автоматический расчёт хэшей при build

**Источник:** Аудит M22, безопасность.md §9

### R-021 — Multi-tenant: schema sharing

**Категория:** Архитектура | **Критичность:** Высокая | **Статус:** 🟡 Мониторинг

**Описание:** Все tenants используют единую PostgreSQL schema `lkfl_platform` с `tenant_id` для изоляции. Один tenant может нагрузить другой через:
- Resource contention (CPU, I/O, connections)
- Long-running queries
- Cache pollution

**Влияние:** Tenant A с высоким трафиком → деградация SLA для tenant B.

**Митигация:**
- Per-tenant connection pool в PgBouncer
- Query timeout per tenant
- Rate limiting per tenant (уже реализовано)
- При >5 крупных tenants: dedicated DB instance

**Источник:** -

### R-027 — Tenant isolation — нет RLS

**Категория:** Безопасность | **Критичность:** Высокая | **Статус:** 🟡 Мониторинг

**Описание:** Изоляция tenant'ов реализована только на уровне application (middleware + FK constraint). Нет Row-Level Security на уровне PostgreSQL.

**Влияние:**
- Прямой доступ к БД bypass'ит application-level изоляцию
- DBA может увидеть данные всех tenants
- Нарушение принципа defense in depth

**Митигация:** PostgreSQL RLS policy (M29+):
```sql
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON users
  USING (tenant_id = current_setting('app.tenant_id'));
```

**Источник:** -

### R-028 — Audit trail mutable

**Категория:** Безопасность | **Критичность:** Высокая | **Статус:** 🟡 Мониторинг

**Описание:** Audit trail таблицы (audit_log) — mutable. DELETE не запрещён на уровне БД. ФСТЭК требует immutable audit trail.

**Влияние:**
- Атакующий с DBA доступом может удалить следы
- Нарушение требований ФСТЭК (audit trail immutable)
- Потеря данных для forensic analysis

**Митигация:**
```sql
-- Trigger: ON DELETE → append to audit_log_history (append-only)
CREATE OR REPLACE FUNCTION prevent_audit_delete() RETURNS trigger AS $$
BEGIN
  RAISE EXCEPTION 'DELETE on audit_log is not allowed';
END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER audit_log_no_delete BEFORE DELETE ON audit_log
  FOR EACH ROW EXECUTE FUNCTION prevent_audit_delete();
```

**Источник:** -

### R-029 — Session management — нет idle timeout

**Категория:** Безопасность | **Критичность:** Средняя | **Статус:** 🟡 Мониторинг

**Описание:** Redis session TTL — 7 дней. Нет idle timeout. Активная сессия живёт 7 дней без ре-аутентификации.

**Влияние:** При краже сессионного cookie → доступ на 7 дней.

**Митигация:**
- Idle timeout: 30 мин без активности → require re-auth
- Absolute timeout: 24 часа → require re-auth
- Session rotation при изменении UA/IP

**Источник:** 152-ФЗ, безопасность.md §12 (Sessions TTL 7 дней)

### R-030 — Admin API — нет IP allowlist

**Категория:** Безопасность | **Критичность:** Высокая | **Статус:** 🟡 Мониторинг

**Описание:** Admin endpoints (`/admin/**`) доступны из любой сети при валидном JWT. Нет IP allowlist для super-admin операций (tenant creation, seed data, deploy trigger).

**Влияние:** При компрометации admin JWT → полный доступ к платформе из любой точки мира.

**Митигация:** IP allowlist в Nginx для `/admin/**`:
```nginx
location /admin/ {
  allow 10.0.0.0/8;      # internal
  allow 192.168.0.0/16;  # corporate VPN
  deny all;
}
```

**Источник:** OWASP A01

### R-036 — Мониторинг — нет security alerting

**Категория:** Архитектура | **Критичность:** Высокая | **Статус:** 🟡 Мониторинг

**Описание:** Текущие alert rules (T2206) покрывают operational метрики (error rate, latency, uptime), но не security events:
- Brute force: >10 failed login attempts per IP
- Unusual access pattern: admin access из нового IP/UA
- Mass data export: >100 records per request
- Rate limit violations: >50 429 responses per minute

**Влияние:** Security incidents не обнаруживаются в real-time.

**Митигация:** Добавить alert rules:
```yaml
- alert: BruteForceDetected
  expr: rate(http_requests_total{code="401", path=~"/api/v1/auth/.*"}[5m]) > 0.5
  for: 2m
  labels:
    severity: critical
- alert: AdminAccessAnomaly
  expr: increase(http_requests_total{path=~"/admin/.*", user_role="super_admin"}[1h]) > 100
  for: 5m
  labels:
    severity: warning
```

**Источник:** -

## Процесс управления рисками

### Когда добавлять риск

1. **Аудит реализации** (hardening, milestone review) — как в M22
2. **Code review** — обнаружение уязвимости/проблемы
3. **ADR** — новый архитектурный риск при принятии решения
4. **Incident postmortem** — риск, приведший к инциденту
5. **Penetration test** — найденные уязвимости
6. **Dependency update** — CVE в зависимости

### Формат нового риска

```markdown
### R-{NNN} — {Название}

**Категория:** {Безопасность | Надёжность | Архитектура | Совместимость | Комплаенс | Performance}
**Критичность:** {Критическая | Высокая | Средняя | Низкая | Информация}
**Статус:** {🔴 Открыт | 🟡 Мониторинг | 🟢 Принят | ✅ Закрыт | ⛔ Не актуален}

**Описание:** {Что за риск и почему он важен}
**Влияние:** {Что произойдёт, если риск реализуется}
**Митигация:** {Как снизить/устранить риск}
**Источник:** {Где обнаружен: аудит M22, ADR-035, пентест и т.д.}
```

### Пересмотр рисков

- **Ежемесячно:** архитектор проверяет статус открытых/мониторинга рисков
- **При завершении вехи:** все риски из audit отчёта добавлены в реестр
- **При изменении архитектуры:** проверить актуальность существующих рисков
