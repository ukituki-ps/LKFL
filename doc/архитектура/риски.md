## Детальное описание

### R-01: Компрометация JWT → меж-tenant доступ

```
Attacker → Подделывает JWT → Bypass tenant resolution → Чтение данных другого tenant'а
```

**Смитагация R-01-S1:**
1. Keycloak signature verification (RS256)
2. Realm-specific JWKS — токен realm A не валидируется JWKS realm B
3. Tenant Resolver middleware — проверяет subdomain → tenant
4. DB FK constraint — `tenant_id` на каждой таблице
5. Audit log — все запросы логируются (user_id, tenant_id, endpoint)

---

### R-02: Tenant resolution bypass

```
Attacker → Пропускает Tenant Resolver → Доступ к API без tenant context
```

**Смитагация R-02-S1:**
1. Tenant Resolver — первый middleware в цепи, нельзя пропустить
2. JWT validation использует tenant-specific JWKS
3. Handler без tenant_id в context → panic (fail-safe)
4. All DB queries require tenant_id

---

### R-03: Keycloak downtime → полная недоступность

```
Keycloak down → Нет JWKS → Нет JWT validation → Все API возвращают 401
```

**Смитагация R-03-S1:**
1. JWKS cache with long TTL (JWKS меняются редко — при key rotation)
2. Redis backup cache для JWKS
3. Graceful degradation: если Keycloak недоступен, использовать cached JWKS
4. Health check: `/readyz` проверяет Keycloak connectivity

---

### R-07: Realm per tenant → operational complexity

Каждый новый tenant требует:
- Новый Keycloak realm
- Новые OIDC clients
- Новые protocol mappers
- Новый JWKS endpoint в cache

**Смитагация:**
1. `infra/keycloak/realm/tenant-template.json` — шаблон realm
2. `infra/keycloak/scripts/create-tenant-realm.sh` — скрипт создания
3. Terraform module для автоматизации
4. Документация: tenant onboarding process (2 часа)

---

### R-08: Dynamic JWKS cache miss → latency

При первом запросе tenant'а JWKS нужно загрузить с Keycloak:

```
First request → Tenant Resolver → Load JWKS (HTTP call) → Parse → Cache → JWT Validate
Latency: +50-200ms для первого запроса tenant'а
```

**Смитагация R-08-S1:**
1. Cache all JWKS at startup (known tenants)
2. Lazy load with background refresh
3. Redis shared cache between replicas
4. JWKS TTL: 1 hour (refresh only on key rotation)

---

### R-09: Single tenant realm compromise → full tenant exposure

Если злоумышленник получает доступ к Keycloak realm конкретного tenant'а:

```
Realm compromise → Create malicious users → Get valid JWT → Full tenant access
```

**Смитагация R-09-S1:**
1. Keycloak admin access — отдельный IAM policy
2. MFA required для admin/hr ролей
3. Audit log на Keycloak admin операции
4. Если realm скомпрометирован → ротация всех ключей realm'а

---

### R-10: Keycloak JWKS endpoint throttling

Если Keycloak JWKS endpoint получает слишком много запросов (много tenant'ов одновременно делают first request):

```
Many tenants → Simultaneous JWKS requests → Keycloak throttled → All JWT validation delayed
```

**Смитагация R-10-S1:**
1. Cache all known tenants JWKS at server startup
2. Rate limit JWKS requests per tenant (max 1/minute)
3. Background refresh (не на request path)
4. Redis shared cache между репликами сервера

---

### R-13: M17 scope creep

M17 охватывает ВСЮ систему авторизации — инфраструктура, backend, frontend, тесты, docker-compose. Высокий риск:
- Занять больше времени, чем планировалось
- Упустить edge-кейсы из-за масштаба
- Заблокить все последующие задачи

**Смитагация R-13-S1 (обновлено после актуализации задач):**
1. M17 разбита на 5 задач с чёткими границами: T1701 (инфра), T1702 (backend auth), T1703 (frontend auth), T1704 (CI/CD), T1705 (Observability, можно отложить)
2. Каждая задача имеет собственный brief.md + plan.yaml — нет ничего вне плана
3. Инкрементальные проверки: после T1701 — `make docker-up`, после T1702 — `go build`, после T1703 — `npm run dev`
4. YAGNI: только то, что нужно для login → /api/v1/me → 200
5. Не делать: admin UI для tenant management, advanced Keycloak flows, custom themes
6. После M17 перепланируем M18+ с учётом реальной скорости

### R-17: Frontend token rotation race condition

```
Request A starts → token expires → Request B refreshes token → Request A sends with old token → 401
```

**Смитагация R-17-S1:**
1. `authorizedFetch` с serial refresh — один refresh на все pending requests
2. Queue mechanism: requests during refresh wait for new token
3. React Query invalidate on token rotation — пересоздание запросов с новым токеном
