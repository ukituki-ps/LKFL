# TagResolver — вычисление тегов пользователя

## TL;DR (для агентов)

> Этот файл описывает **TagResolver** — вычисление тегов пользователя для CEL.
> - **Теги** — не исторические данные, а current-state из профиля: `is_remote`, `is_newbie`, `has_children` и др.
> - **Где используются:** в CELContext для eligibility/gamification правил
> - **Полная архитектура CEL** → `cel-engine.md` строка 111 (CELContext)

> **P1:** заменяет «Backfill Worker» из blind-spot анализа. Теги — не исторические данные, а current-state вычисления из профиля пользователя.
> **Связь:** `cel-engine.md` → `Tags map[string]interface{}` в CELContext. `пакеты-platform.md` → `cel/` package.

---

## Проблема

CELContext содержит `Tags map[string]interface{}` для динамических меток пользователя:
- `tags.is_remote` — работает удалённо
- `tags.is_newbie` — стаж < 3 месяцев
- `tags.pilot_program` — участник пилота
- `tags.is_senior` — стаж >= 5 лет

**Вопрос:** откуда берутся эти теги?

### Что это НЕ является

- **НЕ отдельный сервис** — теги вычисляются из данных, которые уже есть в `user/`
- **НЕ историческая таблица** — `tags.is_senior` = current state, не «становился senior в 2024»
- **НЕ требует backfill** — при запуске системы теги мгновенно вычисляются из HR-данных

---

## Решение — TagResolver (in-process)

TagResolver — функция внутри `cel/` пакета, которая собирает теги из профиля пользователя **при каждом запросе CELContext**.

```
FlowEngine.Activate()
  → собирает CELContext
    → TagResolver.Resolve(user, tenant)
      → { is_remote: true, is_newbie: false, is_senior: true, pilot_program: false }
    → CELEvaluator.Evaluate(celExpr, CELContext)
```

### API

```go
// tag_resolver.go — внутри platform/internal/cel/

type TagResolver struct {
    userRepo UserRepository  // для чтения профиля
    cache    *redis.Client   // Redis `cel:` prefix (TTL 1h)
}

// Resolve вычисляет все теги для пользователя
func (r *TagResolver) Resolve(ctx context.Context, user *User) map[string]interface{}

// ResolveForBatch — оптимизированная версия для batch-операций
func (r *TagResolver) ResolveForBatch(ctx context.Context, users []User) []map[string]interface{}
```

### Каталог тегов

| Тег | Источник | Формула | Обновление |
|-----|----------|---------|-----------|
| `is_remote` | `user.location == "remote"` | в реальном времени | При изменении location в HR |
| `is_newbie` | `user.years_of_service < 0.25` | в реальном времени | Автоматически |
| `is_senior` | `user.years_of_service >= 5` | в реальном времени | Автоматически |
| `tenure_1y` | `user.years_of_service >= 1` | в реальном времени | Автоматически |
| `tenure_3y` | `user.years_of_service >= 3` | в реальном времени | Автоматически |
| `tenure_5y` | `user.years_of_service >= 5` | в реальном времени | Автоматически |
| `has_family_in_dms` | `consents WHERE provider='dms' AND has_dependents` | при запросе | При изменении consent |
| `pilot_program` | `user_engagements WHERE offer_id IN (pilot_offers)` | при запросе | При подключении к пилоту |
| `interest:*` | `user_survey_attributes` (survey-ответы) | при запросе | При completion опроса (TagMapper.MapSurveyAnswers) |
| `sport_intensity` | `user_survey_attributes` (survey-ответы) | при запросе | При completion опроса (TagMapper.MapSurveyAnswers) |

> **Важно:** profile-теги — **вычисляемые**, не хранимые. Нет таблицы `user_tags`. Теги выводятся из первичных данных (user profile, consents, engagements).
> **M13 T1301 (спроектировано):** survey-теги — исключение: они хранимые (таблица `user_survey_attributes`), но агрегируются как вычисляемые через MAX(weight) GROUP BY tag_key. Реализация в M14.

### Кэширование

Теги кэшируются в Redis (prefix `cel:`, совместно с CEL programs):

```
cel:tag:{tenant_id}:{user_id} → JSON { is_remote: true, is_senior: true, ... }
TTL: 1h
```

> ⚠️ **Было:** «Redis DB 4» — numbered DB. **После M12:** key prefixes (`cel:`) вместо numbered DB. См. `schema.md § Redis layout`.

**Invalidate при:**
- Обновление user profile (grade, location, department) → `DEL tag_cache:{tenant}:{user}`
- Изменение consent → invalidate user's tag cache
- HR sync batch → invalidate all tags для изменённых пользователей
- **M13 T1301 (planned, M14):** Survey completion → TagMapper.MapSurveyAnswers → InvalidateCache(user)

### Survey-теги (M13 T1301 — спроектировано, M14 — реализация)

Survey-теги отличаются от profile-тегов:
- **Источник:** `user_survey_attributes` (таблица БД, не вычисляется live из профиля)
- **Агрегация:** `MAX(weight) GROUP BY tag_key` — перекрёстные опросы сохраняют максимальный вес
- **Invalidate:** при MapSurveyAnswers (survey completion), при изменении ответа внутри survey_offer_id
- **Совместимость:** возвращают `map[string]interface{}` — напрямую merge в CELContext.Tags

```
TagResolver.Resolve(ctx, user):
  tags := computeFromProfile(user)        // is_remote, is_senior, ...
  surveyTags := AggregateSurveyTags(ctx, tenantId, userId)  // interest:sport, ...
  for k, v := range surveyTags { tags[k] = v }
  return tags  // → CELContext.Tags
```

## Почему не хранимые теги

| Подход | Плюсы | Минусы |
|--------|-------|--------|
| **TagResolver (выбранный)** | Нет денормализации, всегда актуально, нет backfill | +1 query при каждом CELContext |
| Хранимая таблица `user_tags` | Быстрый read | Денормализация, race condition при update, нужен backfill при запуске, eventual consistency |

**Аргумент за TagResolver:** теги — это view over user data. Они не являются самостоятельной сущностью. Кэширование в Redis даёт O(1) после первого вычисления.

## Интеграция с CELContext

```go
// context.go — внутри platform/internal/cel/

func BuildCELContext(ctx context.Context, user *User, offer *Offer, extra map[string]any) *CELContext {
    tags := tagResolver.Resolve(ctx, user)

    return &CELContext{
        User: struct { ... }{
            Grade:          user.Grade,
            YearsOfService: user.YearsOfService,
            // ...
        },
        Tags: tags,  // ← теги встроены в контекст
        Benefit: struct { ... }{
            Category: offer.Category,
            Cost:     offer.Cost,
        },
        // ...
    }
}
```

## Геймификация и теги

Теги геймификации (`game.achievements`, `game.loyalty_level`) **не входят** в TagResolver. Они управляются `gamification/` пакетом (ADR-023):

| Тег | Источник | Где |
|-----|----------|-----|
| `game.achievements` | `achievement_grants` | `gamification/` |
| `game.loyalty_level` | `user_loyalty_levels` (active) | `gamification/` |
| `game.engagement_count` | `COUNT(user_engagements WHERE completed)` | `gamification/` |
| `game.engagement_by_category` | `GROUP BY category` | `gamification/` |

При сборе CELContext для геймификации:
```go
tags := tagResolver.Resolve(ctx, user)          // базовые теги
gameCtx := gamification.BuildGameContext(ctx, user)  // game.* теги
celCtx.Tags["game"] = gameCtx  // объединение
```
