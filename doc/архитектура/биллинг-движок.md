## Роль движка

Биллинговый движок — **сердце платформы**. Он управляет виртуальной экономикой льгот:
- начисляет средства (баллы / рубли / кредиты)
- списывает при подключении льгот
- отслеживает периоды распределения
- сжигает неиспользованные остатки
- обеспечивает финансовую точность (ACID на каждую транзакцию)

Валюта — **конфигурируется per tenant**. Может быть «баллы», «рубли», «кредиты» или смешанная модель.

## Billing Rule — универсальное правило

Каждая финансовая операция в системе — результат выполнения правила. Правило — конфигурируемый объект, не код.

### Структура правила

```yaml
rule:
  id: uuid
  tenant_id: uuid
  name: "Ежемесячное начисление по грейдам"
  description: "Начисление 1000 баллов сотрудникам grade-A"

  # Направление
  direction: credit | debit

  # Сумма
  amount:
    fixed: 1000                              # фиксированная сумма
    # ИЛИ
    expression: "grade_table[user.grade]"    # формула (таблица grade→amount)
    # ИЛИ
    expression: "engagement_offer_cost * 0.7"         # динамическая (от контекста)

  # Категория
  category: "Общая" | "ДМС" | "Фитнес" | ...  # конфигурируемая строка

  # Частота
  frequency: one-time | monthly | quarterly | yearly

  # Триггер
  trigger:
    type: cron | event | manual
    cron: "0 0 1 * *"                         # для cron: schedule
    event: "engagement_debit"                 # для event: какой event
    event_filter: "category=dms"              # опционально: фильтр по event

  # Условие применения (optional)
  # **ADR-021:** condition → CEL expression. Старый YAML-array `condition` удалён после миграции.
  # HR вводит текст на русском → LLM генерирует CEL → система валидирует → сохраняет.
  condition_cel: "user.grade in ['A', 'B'] && user.years_of_service >= 3 && user.has_children"
  condition_source: "только грейды A и B, стаж от 3 лет, есть дети"
  condition_llm_model: "qwen3.6:27b"
  condition_llm_version: "v1.0.0"

  # Приоритет (для разрешения конфликтов)
  priority: 10

  # Статус
  status: active | paused

  # Мета
  created_by: user_id
  created_at: timestamp
  effective_from: date    # дата вступления в силу
  effective_to: date      # дата окончания (optional)
```

## Движок правил — как работает

> **ADR-021:** условие `condition_cel` — CEL expression. Evaluation через `cel/` package (cel-go).
> Старый YAML-array `condition` → удалён после миграции Фаза A.

### Шаг 1: Сбор контекста

При срабатывании триггера движок собирает CELContext:
```
CELContext = {
  user: { grade, years_of_service, has_children, status, tenant_id, user_id },
  tags: { is_remote, is_newbie, pilot_program, ... },
  benefit: { category, cost },
  date: { today },
  context: { last_engagement, engagement_active, engagement_provider, ... },
  events: { type, engagement_offer_id, engagement_offer_cost, provider, ... },
  period: { start, end },
  balance: { total }
}
```

### Шаг 2: Фильтрация правил (DB + CEL)

```
matching_rules = rules WHERE
  tenant_id = context.user.tenant_id
  AND status = 'active'
  AND effective_from <= now()
  AND (effective_to IS NULL OR effective_to >= now())
  AND trigger_matches(context)

FOR rule IN matching_rules:
  result = CEL_Evaluator.Evaluate(rule.condition_cel, CELContext)
  IF result == true:
    rule.is_applicable = true
```

### Step 2.1 — LLM fallback (для новых правил из Admin UI)

Admin отправляет `condition_source` (русский текст):
```
IF rule.condition_cel is EMPTY:
  cel = CEL_Generator.Generate(rule.condition_source)
  CEL_Validator.Validate(cel)
  rule.condition_cel = cel
```

### Шаг 3: Применение по приоритету

```
FOR rule IN matching_rules ORDER BY priority ASC:
  amount = evaluate(rule.amount, context)
  tx = create_transaction(user_id, rule, amount)
  IF tx.valid:
    apply(tx)
    emit_event(tx)
  ELSE:
    log_error(tx)
    notify_admin(tx)
```

### Шаг 4: ACID-гарантии

Каждая транзакция — в рамках DB-транзакции:
```sql
BEGIN;
  UPDATE accounts SET balance = balance + :amount WHERE user_id = :uid;
  INSERT INTO transactions (user_id, rule_id, type, amount, category, context)
  VALUES (:uid, :rid, :type, :amount, :category, :ctx);
COMMIT;
```

## Периоды распределения

Период — временное окно, в которое сотрудник выбирает льготы.

```
Период:
  id: uuid
  name: "Q3 2026"
  start_date: 2026-07-01
  end_date: 2026-09-30
  status: draft | open | closed | expired

  # Настройки
  auto_credit_on_open: true
  credit_amount: 5000
  burn_unspent: true          # сжигать остаток в конце
  burn_warning_days: 30       # напоминание за N дней
```

### Lifecycle периода

```
Draft → Open → Closed → Expired

Draft: HR настроил, но ещё не открыл
Open:   Сотрудники видят каталог, могут подключать льготы
Closed: Новые подключения запрещены, старые активны
Expired: Период завершён, остатки сгорели, архив
```

### Сгорание баллов

```
При переходе Closed → Expired:
  FOR each user WITH balance > 0:
    create_transaction(user_id, type=burn, amount=balance, category="Сгорание")
    update_accounts SET balance = 0
    send_notification("Ваши N баллов сгорели")
```

## Транзакция

```
Transaction:
  id: uuid
  user_id: uuid
  rule_id: uuid              # ссылка на правило, которое её создало
  type: credit | debit | burn | revert
  amount: decimal
  category: string
  source: string             # описание источника (для UI)
  status: created | frozen | confirmed | cancelled | error
  idempotency_key: string    # ⚠️ P0: уникальный ключ идемпотентности (sha256 hash)
  context: JSONB             # snapshot контекста (для аудита)
  created_at: timestamp
```

### Идемпотентность (P0 — критично)

> **M12:** NATS JetStream удалён. Дублирование возможно только при重试 HTTP-вызова от Platform → Billing в рамках одной PG транзакции. ACID гарантирует нет двойного списания.

**Механизм:**

```
Platform: idempotency_key = sha256(userId + ":" + offerId + ":" + periodId)
          billing.BillingService.Debit(ctx, userId, offerId, periodId, idempotency_key)

Billing:  INSERT INTO transactions (idempotency_key, user_id, ...)
          VALUES (:key, :uid, ...)
          ON CONFLICT (idempotency_key) DO NOTHING;

          IF rows_affected == 0:
            log_warn("duplicate call ignored, idempotency_key=%s", key)
            return existing_transaction
          ELSE:
            apply_and_commit()
```

**SQL:**

```sql
ALTER TABLE transactions
  ADD COLUMN idempotency_key VARCHAR(64),
  ADD CONSTRAINT uq_transactions_idempotency_key UNIQUE (idempotency_key);

-- Индекс для быстрого поиска дубликатов
CREATE INDEX idx_transactions_idempotency_key ON transactions(idempotency_key);
```

**Важно:**
- `ON CONFLICT DO NOTHING` — атомарная операция на уровне PostgreSQL, нет race condition
- Ключ детерминированный (hash), не UUID — Platform может вычислить его повторно при перезапуске
- M12 (direct call): единая PG транзакция → дублирование_delivery невозможно; идемпотентность остаётся для защиты от重试 клиентской стороны
- Query-операции не требуют идемпотентности — они idempotent by nature (read-only)

### Revert (отмена транзакции)

```
Реверт создаёт компенсационную транзакцию:
  IF original.type == credit:
    new.type = debit, new.amount = -original.amount
  IF original.type == debit:
    new.type = credit, new.amount = -original.amount

Пример: HR отклонил заявку → баллы возвращены (J20)
Пример: Провайдер отменил активацию → баллы возвращены (T05)
```

## Интеграция с Platform (M12)

Биллинг — `internal/billing/` внутри lkfl-server. Общение через direct Go call (был NATS JetStream pre-M12).

> **M12 ADR-024:** NATS subjects `billing.*` удалены. Все вызовы → `billing.BillingService.*()` direct call.

```go
// Direct call из Platform → internal/billing/
billing.BillingService.Credit(ctx, userId, amount)             // начислить X баллов
billing.BillingService.DebitReserve(ctx, userId, offerId)     // заморозить при pending
billing.BillingService.DebitConfirm(ctx, userId, offerId)     // подтвердить debit
billing.BillingService.DebitReverse(ctx, userId, txId)        // возврат при failed
billing.BillingService.QueryBalance(ctx, userId)              // запросить баланс
billing.BillingService.QueryTransactions(ctx, userId)         // история транзакций
```

> **NATS subjects (pre-M12, исторически):**
> - `billing.credit`, `billing.debit.reserve`, `billing.debit.confirm` → теперь `BillingService.Credit/Debit*`
> - `billing.balance.query`, `billing.transactions.query` → теперь `BillingService.Query*`

## Payroll — 1C интеграция (M11 T1103, ADR-027)

> **Перенесён из `integrations/1c/` в `internal/billing/payroll/`.** Бухгалтерия (удержание из ЗП) belongs к finance domain — ближе к транзакциям, audit trail, billing rules.

### Назначение

Передача заявлений на удержание из заработной платы в 1С. Отличается от payment-gateway (PCI DSS карточные платежи): это бухгалтерская операция, не платёжная.

### API

| Метод | Описание |
|-------|---------|
| `SubmitPayroll(stmt *PayrollStatement)` | Передать заявление на удержание ЗП в 1C |
| `ListPayrollStatements(tenantId string) []PayrollStatement` | Список всех заявлений tenant'а |
| `GetPayrollStatus(id string) (*PayrollStatus, error)` | Статус конкретного заявления |

### Flow (M12: был NATS, теперь direct call)

```
Platform `engagement/flow/FlowEngine.Activate()` → `billing/payroll.SubmitPayroll(stmt)`
  → 1C REST API (POST /api/payroll/submit)
  → результат → audit trail (PostgreSQL)
```

> **M11 T1104:** namespace rename: было `integration.payroll.submit` → `finance.payroll.submit`.
> **M12:** NATS subject удалён. `engagement/flow/` → direct call `billing.PayrollEngine.SubmitPayroll(stmt)`.

### Flow с payments

При DMS upgrade → выбор оплаты "удержание из ЗП":
1. Wizard → engagement activation (Platform)
2. `engagement/flow/FlowEngine.Activate()` → `billing/payroll.SubmitPayroll(stmt)` call напрямую
3. Billing `payroll.SubmitPayroll(stmt)` → 1C REST API
4. 1C возвращает `statementId` → Billing сохраняет в audit trail
5. Если `payments/` канал нужен (доплата сотрудника): `payments.PaymentGateway.CapturePayroll(stmt)`

### Зависимости

- 1C REST API — прямой вызов
- **M12:** `engagement/flow/` → direct call `billing.PayrollEngine.SubmitPayroll(stmt)` (был NATS consumer `finance.payroll.submit`, ADR-027)
- `db/` — PostgreSQL (payroll_statements table)

## Управление правилами (API)

> **M12:** endpoints перенесены в `/api/v1/billing/` (было `/billing/v1/`). Billing — `internal/billing/`, не отдельный сервис.
> **Синхронизация:** endpoints совпадают с `спецификация/api.md → Billing`. Pause/resume управляются через PUT `status: paused/active`.

Управленческий API для HR / менеджера (через Nginx → lkfl-server:8080):

| Метод | Endpoint | Описание |
|-------|----------|----------|
| GET | `/api/v1/billing/rules` | Список правил (с фильтрами) |
| POST | `/api/v1/billing/rules` | Создание правила |
| GET | `/api/v1/billing/rules/:id` | Детали правила |
| PUT | `/api/v1/billing/rules/:id` | Редактирование (вкл. pause/resume через `status: paused/active`) |
| DELETE | `/api/v1/billing/rules/:id` | Удаление (soft delete) |
| POST | `/api/v1/billing/periods/:id/activate` | Открыть период распределения |
| POST | `/api/v1/billing/periods/:id/expire` | Закрыть период → сгорание остатков → остановка распределения |
| GET | `/api/v1/billing/periods/current` | Текущий активный период |

**8 endpoints.** Операционные вызовы (credit, debit, balance, transactions) — direct Go call `billing.BillingService.*()`, не HTTP.
