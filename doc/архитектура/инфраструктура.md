## Баги dev-стенда

### Б-001: Healthcheck в Dockerfile — все контейнеры unhealthy

| Поле | Значение |
|------|----------|
| **Дата обнаружения** | 2026-05-27 |
| **Симптом** | Все контейнеры (server, proxy, keycloak) в статусе `unhealthy` |
| **Причина** | `Dockerfile` содержал `HEALTHCHECK CMD ["wget", ...]`, но `distroless/base-debian12` не включает `wget` (нет shell, нет package manager) |
| **Влияние** | Docker не мог определить готовность сервисов; `depends_on: service_healthy` не работал |

#### Решение

```diff
 # Dockerfile — server stage
-FROM gcr.io/distroless/base-debian12 AS server
+FROM gcr.io/distroless/base-debian12 AS server
 # ...
-HEALTHCHECK --interval=10s --timeout=5s --retries=3 \
-    CMD ["wget", "-q", "--spider", "http://localhost:8080/healthz"]
+# No HEALTHCHECK — distroless has no shell/wget.
+# Healthcheck managed externally via docker-compose nginx upstream.
```

Healthcheck перенесён в `docker-compose.server.yml` на уровне nginx:
```yaml
nginx:
  healthcheck:
    test: ["CMD-SHELL", "curl -f http://localhost:8080/healthz || exit 1"]
```

#### Урок

> **Distroless образы не имеют shell/wget/curl.** Healthcheck только через внешний nginx или отдельный healthcheck-образ.

### Б-002: Keycloak генерирует `http://` URL в формах логина

| Поле | Значение |
|------|----------|
| **Дата обнаружения** | 2026-05-27 |
| **Симптом** | Страница логина Keycloak содержит `http://dev.april.ukituki.tech` вместо `https://` → браузер блокирует mixed content |
| **Причина** | 1. `KC_HTTP_ENABLED: "true"` заставлял Keycloak генерировать `http://` URL. 2. `proxy=passthrough` в `keycloak.conf` — deprecated в Keycloak 25.x. 3. Nginx передавал `X-Forwarded-Proto: $scheme` → `http` (nginx общается с Keycloak по HTTP локально). 4. Keycloak 25.x использует `hostname:v2` — старые настройки v1 игнорируются |
| **Влияние** | Login flow не работал — браузер блокировал переход на `http://` |

#### Решение

**1. Кастомный `infra/keycloak/keycloak.conf`:**
```properties
hostname=dev.april.ukituki.tech
hostname-strict=false
proxy-headers=xforwarded
http-enabled=false
```

**2. Nginx — хардкод `X-Forwarded-Proto: https` для Keycloak routes:**
```nginx
location /protocol {
    proxy_pass http://keycloak;
    proxy_set_header X-Forwarded-Proto https;  # было: $scheme
    proxy_set_header X-Forwarded-Port 443;
    # ...
}
```

#### Урок

> **Keycloak за reverse proxy требует явного указания HTTPS.** `$scheme` в nginx = `http` для внутренних соединений. Keycloak 25.x использует `hostname:v2` — старые `proxy=...` настройки не работают.

### Б-003: 401 на `/api/v1/auth/callback` — потеря state в Redis

| Поле | Значение |
|------|----------|
| **Дата обнаружения** | 2026-05-27 |
| **Симптом** | `GET /api/v1/auth/callback` → 401 `invalid or expired state parameter` |
| **Причина** | Redis терял данные при перезапусках контейнеров. State сохранялся при login redirect, но контейнер Redis перезапускался (из-за исправления healthcheck/HTTPS) → state исчезал. Код auth handler корректен (state key совпадает login ↔ callback, TTL 10 мин, формат `auth:state:{hash}`). |
| **Влияние** | Login flow не завершался — бесконечный цикл login → 401 → login |

#### Решение

**1. Redis persistence в `docker-compose.server.yml`:**
```yaml
redis:
  command: >
    redis-server
    --appendonly yes
    --appendfsync everysec
    --save 60 1
    --maxmemory 256mb
    --maxmemory-policy allkeys-lru
    --bind 0.0.0.0
```
- `--appendonly yes` — AOF journal
- `--appendfsync everysec` — fsync каждую секунду (баланс производительность/безопасность)
- `--save 60 1` — RDB snapshot если ≥1 ключ изменён за 60 сек

**2. Graceful error handler в `backend/internal/auth/handler.go`:**
```go
// Было: 401 Bad Request → бесконечный цикл
// Стало: 410 Gone для API, 302 → /login для browser
if isBrowserRequest(r) {
    http.Redirect(w, r, "/login?error=expired_session", http.StatusFound)
    return
}
shhttp.WriteJSONError(w, http.StatusGone, "session expired, please login again")
```

**3. Frontend anti-loop в `frontend/src/pages/Callback.tsx`:**
- Счётчик попыток в `sessionStorage`
- При ≥2 неудач → logout из Keycloak + полная очистка сессии
- Обработка 410 → retry с `?retry=1` → `prompt=login` в Keycloak

**4. Login retry с `prompt=login` в `backend/internal/auth/handler.go`:**
```go
if r.URL.Query().Get("retry") == "1" {
    prompt = "login"  // форсируем перелогин в Keycloak
}
```

#### Урок

> **Redis на dev-стенде требует persistence.** Без AOF/RDB данные теряются при restart. Auth callback flow (state + PKCE) — критический путь: state должен выжить перезапуск.

### Б-004: 500 на callback — каскад ошибок БД

| Поле | Значение |
|------|----------|
| **Дата обнаружения** | 2026-05-27 |
| **Симптом** | `GET /api/v1/auth/callback?code=...&state=...` → 500 Internal Server Error (154B body) |
| **Причина** | 4 подпричины, обнаруженные последовательно |
| **Влияние** | Login flow не завершался даже при валидном state и code |

#### Подпричина 4A: БД пуста — миграции не применены

```
psql \dt → 0 таблиц
```

Миграции `20260526100000_init.sql` и `20260526110000_tenants.sql` существовали в `migrations/`, но не были применены. Таблица `users` вообще отсутствовала (миграция не была создана).

**Решение:**
- Применены init + tenants миграции
- Создана `migrations/20260526120000_users.sql` по схеме из `doc/архитектура/schema.md`

#### Подпричина 4B: Tenant не в БД

```sql
SELECT * FROM tenants; → 0 rows
```

`CreateOrUpdateUser` с `tenant_id = uuid.Nil` → FK constraint violation.

**Решение:**
- Seed tenant `sdek` + brand config
- Добавлен tenant resolver в auth service (`backend/internal/auth/service.go`):
  ```go
  // Fallback tenant resolution для auth callback (без tenant middleware)
  tenantResolver := tenant.NewHostResolver(tenantService, redisClient, appMetrics)
  authService.WithTenantResolver(tenantResolver)
  // Извлекаем slug из issuer URL: .../realms/lkfl-sdek → sdek
  parts := strings.Split(cfg.KeycloakIssuer, "/")
  for _, p := range parts {
      if len(p) > 5 && p[:5] == "lkfl-" {
          authService.SetDefaultTenantSlug(p[5:])
          break
      }
  }
  ```

#### Подпричина 4C: Колонка `keycloak_sub` ≠ `keycloak_user_id`

| Место | Имя колонки |
|-------|-------------|
| Миграция `20260526120000_users.sql` | `keycloak_user_id` ✅ |
| Repository `backend/internal/user/repository.go` | `keycloak_sub` ❌ |

**Решение:** 9 замен `keycloak_sub` → `keycloak_user_id` в `repository.go`.

#### Подпричина 4D: Колонка `metadata` отсутствует

Repository INSERT/SELECT включает `metadata`, но миграция не создавала эту колонку.

**Решение:**
```sql
ALTER TABLE lkfl_platform.users ADD COLUMN metadata jsonb DEFAULT '{}';
```

Добавлена в миграцию `20260526120000_users.sql`.

#### Урок

> **Миграции и код должны синхронизироваться.** Разрешение колонок между migration SQL и Go repository — частый источник 500. Чек-лист: schema.md → migration → repository → model.

### Б-005: OIDC issuer mismatch при рестарте сервера

| Поле | Значение |
|------|----------|
| **Дата обнаружения** | 2026-05-27 |
| **Симптом** | Сервер не стартует — 30 попыток OIDC discovery → crash → restart loop |
| **Причина** | `KEYCLOAK_ISSUER: http://keycloak:8080/realms/lkfl-sdek` → Keycloak возвращал `https://dev.april.ukituki.tech/realms/lkfl-sdek` (из-за `keycloak.conf` hostname) → issuer mismatch |
| **Влияние** | Сервер не запускался |

#### Решение

```diff
 # docker-compose.server.yml
  KEYCLOAK_ISSUER: https://dev.april.ukituki.tech/realms/lkfl-sdek
- KEYCLOAK_ISSUER: http://keycloak:8080/realms/lkfl-sdek
```

#### Урок

> **KEYCLOAK_ISSUER должен быть публичным URL**, совпадающим с `hostname` в `keycloak.conf`. Внутренний DNS (`keycloak:8080`) не работает при proxy-режиме.

### Б-006: Docker build без `--target` — proxy вместо server

| Поле | Значение |
|------|----------|
| **Дата обнаружения** | 2026-05-27 |
| **Симптом** | `lkfl-server` возвращает 502 Bad Gateway. Nginx upstream `lkfl-server:8080` → connection refused. В логах: `integration-proxy stub listening on :8090` |
| **Причина** | `docker build -t lkfl-server:latest -f Dockerfile .` (без `--target`) собирает **последний** stage Dockerfile — `proxy`. Контейнер запускает `/app/proxy` на порту 8090 вместо `/app/server` на 8080 |
| **Влияние** | Полная недоступность API. Критический баг деплоя — все endpoint'ы возвращали 502. |

#### Решение

```bash
# Правильно:
docker build --target server -t lkfl-server:latest -f Dockerfile .
docker build --target proxy -t lkfl-integration-proxy:latest -f Dockerfile .

# НЕПРАВИЛЬНО (собирает proxy вместо server!):
docker build -t lkfl-server:latest -f Dockerfile .
```

Dockerfile multi-stage:
```
Stage 4: server (lkfl-server)
Stage 5: proxy (lkfl-integration-proxy) ← последний по умолчанию
```

#### Урок

> **Multi-stage Dockerfile без `--target` = последний stage.** Всегда указывать `--target` или переставить server stage последним.

## Б-007: Стаб-цикл login → dashboard → logout (27 мая 2026)

**Симптом:** После логина пользователь видит меню, но его выкидывает обратно на login.
Цикл повторяется бесконечно.

**Корневые причины (3 бага):**

### Б-007A: Access token вместо ID token в callback

`LoginCallback` возвращал `tokenSet.AccessToken`, а `JWTMiddleware` использует
`oidc.IDTokenVerifier`, который верифицирует только ID токены. Access token
не проходит верификацию → 401 → `clearAuth()` → редирект на login.

**Фикс:** `handler.go` — `tokenSet.AccessToken` → `tokenSet.IDToken`.

### Б-007B: Claims pointer vs value в context

`ExtractClaims` возвращает `*Claims`, но `UserIDFromContext`/`RolesFromContext`
ожидают `Claims` (value). Type assertion `ctx.Value(ClaimsKey).(Claims)` падает
при pointer → `keycloakSub == ""` → 401 в `userHandler.Me`.

**Фикс:** `middleware.go` — `context.WithValue(ctx, ClaimsKey, *claims)` (dereference).

### Б-007C: Frontend не шлёт Accept: application/json

`Callback.tsx` делал `fetch('/api/v1/auth/callback')` без заголовка `Accept`.
Браузер шлёт `Accept: */*`, что `isBrowserRequest()` мог интерпретировать как
браузерный запрос → 302 redirect вместо JSON response.

**Фикс:** `Callback.tsx` — `fetch(..., { headers: { 'Accept': 'application/json' } })`.

## Б-008: Tenant not found на dev-стенде (27 мая 2026)

**Симптом:** После успешного callback (200, roles=[admin]) запрос к `/api/v1/users/me`
возвращает 401 `tenant not found`.

**Причина:** `HostResolver.Resolve()` проверял `X-Tenant-ID` header только для
localhost/www хостов. На dev-стенде `Host: dev.april.ukituki.tech` → slug `dev`.
Tenant `dev` отсутствует в БД (есть только `sdek`). External nginx на `serverPr01`
не ставит `X-Tenant-ID`.

**Фикс:**
1. `JWTMiddleware` извлекает slug из issuer claim (`iss: https://.../realms/lkfl-sdek`
   → slug `sdek`) и ставит `X-Tenant-ID: sdek` если header отсутствует.
2. `HostResolver.Resolve()` проверяет `X-Tenant-ID` **до** host-based resolution
   (приоритет header над subdomain).

**Изменённые файлы:**
- `shared/pkg/auth/middleware.go` — `extractTenantSlug(issuer)` + `r.Header.Set("X-Tenant-ID", slug)`
- `shared/pkg/auth/claims.go` — поле `Issuer string` в `Claims`
- `internal/tenant/middleware.go` — `X-Tenant-ID` priority check
- `infra/nginx/server/default.conf` — `X-Tenant-ID: sdek` для `/api` и `/admin`

**Дополнительно:** `authStore.ts` — persist token в `localStorage` + restore при
инициализации (выживание hard refresh без потери сессии).
