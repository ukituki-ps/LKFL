## Краткое описание

Проект «Платформа гибких льгот» — monorepo с двумя бинарниками и React SPA. Один `go.mod`, один Dockerfile (multi-stage). Модульная изоляция обеспечена через 14 internal бизнес-пакетов монолита + `tenant/` (системный) + `api/` router + `integrationclient/` (gRPC client к proxy) с Go-интерфейсами. Integration Proxy — отдельный бинарник для I/O boundary.

> **M12:** Переход на modular monolith. Было 4 Go-сервиса (platform, billing, integrations, payment-gateway) с NATS JetStream. Стало: один бинарник, 15 business-пакетов + tenant/ + api/ router (17 internal-пакетов), Go function calls вместо NATS. [ADR-024](./adr/024-modular-monolith.md)
> **M16:** Вынос `integrations/` в отдельный бинарник `lkfl-integration-proxy`. Монолит общается с proxy через gRPC на localhost. Proxy — I/O boundary, не business module. [ADR-035](./adr/035-integration-proxy.md)

## lkfl-server — монолит (бизнес-логика)

**Назначение.** HTTP API для фронтенда и worker фоновых задач: пользователи, каталог энгейджментов, документы, поддержка, согласия ПДн, нотификации, рекомендации, биллинг, платежи.

**Точка входа:** `cmd/server/main.go` (HTTP) + `cmd/worker/main.go` (Asynq background jobs).

> **M06:** Platform разбита на internal пакеты.
> **M07-T:** eligibility, compliance, admin handlers разделение.
> **M10 T1002:** LLM Proxy слит как `internal/llm/`.
> **M12:** биллинг, integrations, payments → internal пакеты. NATS → Go function calls.
> **M16:** `integrations/` → перенесён в `integration-proxy/adapters/` (ADR-035). Добавлен `integrationclient/` — gRPC client к proxy.
> **Итого:** 14 бизнес-пакетов + tenant/ (системный) + api/ (router) + integrationclient/ (gRPC client) = 16 internal-пакетов монолита.
> Детали: [`пакеты-platform.md`](./пакеты-platform.md). [ADR-024](./adr/024-modular-monolith.md). [ADR-035](./adr/035-integration-proxy.md)

## lkfl-integration-proxy — I/O boundary (M16)

**Назначение.** Gateway к внешним провайдерам льгот: адаптеры, circuit breaker, retry/timeout, webhook receiver, credential store. Общается с монолитом через gRPC на localhost.

**Точка входа:** `cmd/integration-proxy/main.go` (gRPC :8090 + HTTP webhooks :8091).

> **M16:** Вынос из `internal/integrations/` в отдельный бинарник. Fault isolation, credential isolation, goroutine safety. [ADR-035](./adr/035-integration-proxy.md)

### Структура proxy

```
integration-proxy/
├── adapters/                   # ProviderAdapter реализации (11 провайдеров)
├── circuitbreaker/             # Circuit breaker per provider
├── webhook/                    # Webhook receiver + verifier
├── grpc/                       # gRPC server + generated code

```

### Контракты между пакетами

> **M12:** было NATS (runtime string subjects, no type safety) → стало Go-интерфейсы (compile-time validation).
>
> ⚠️ **Canonical source:** полные сигнатуры публичных API всех internal-пакетов — в [`пакеты-platform.md`](./пакеты-platform.md). Здесь — краткое резюме межмодульных вызовов.

| Вызывающий | Цель | Канал | Canonical |
|------|-----|------|-|
| `engagement/flow/` → `billing/` | `Credit()`, `Debit()`, `GetBalance()`, `Revert()` | direct Go call | [`пакеты-platform.md § billing/`](./пакеты-platform.md#пакет-internalbilling--m12-бывший-отдельный-сервис) |
| `engagement/flow/` → `integrationclient/` | `Activate()`, `Deactivate()`, `GetProviderStatus()`, `SyncCatalog()` | **gRPC** (localhost :8090) → proxy | [ADR-035](./adr/035-integration-proxy.md) |
| `engagement/flow/` → `payments/` | `Authorize()`, `Capture()`, `Void()` | direct Go call | [`пакеты-platform.md § payments/`](./пакеты-platform.md#пакет-internalpayments--m12-бывший-отдельный-сервис) |
| `admin/` (API) → `billing/` | `EvaluateRule()`, `FilterRules()`, `Activate()`, `Expire()` | direct Go call | то же |
| `admin/` (API) → `integrationclient/` | `Health()`, `Status()`, `ListProviders()`, `TriggerSync()` | **gRPC** (localhost :8090) → proxy | [ADR-035](./adr/035-integration-proxy.md) |

> **Было NATS:** `billing.credit`, `billing.debit`, `billing.balance.*`, `billing.transactions.*`, `provider.engagement.activate`, `payment.authorize`. **M12:** direct Go call для всех внутренних вызовов. **M16:** вызовы к провайдерам → gRPC (mono ↔ proxy на localhost).

### Keycloak — Identity Provider

Keycloak — центральный компонент безопасности и аутентификации. Все пользователи проходят через Keycloak.

**Роль в системе:**
- **Центральный IdP** — единая точка входа для lkfl-server + Frontend
- **OIDC** — JWT-токены для backend (валидация через `go-oidc`)
- **SSO на внешние сервисы** — Keycloak Identity Broker: Keycloak ↔ провайдер (OIDC/SAML redirect)
- **Multi-tenancy** — realm per tenant (или group-based)
- **Role management** — `employee`, `hr`, `catalog_manager`, `admin`, `integration_admin`
- **Password policy** — сложность, история, ротация (90 дней)
- **ФСТЭК** — on-premise deploy

**Интеграция:**

| Компонент | Как общается | Что получает |
|-----|-------|-------|
| Frontend | Keycloak JS adapter | Token, refresh, login/logout |
| lkfl-server | `go-oidc` client | JWT validation, user info, roles |

**Конфигурация:**

| Параметр | Значение |
|-----|-|
| Realm | `lkfl-{tenant}` (per tenant) |
| Clients | `lkfl-server`, `lkfl-frontend`, `lkfl-keycloak` |
| Roles | `employee`, `hr`, `catalog_manager`, `admin`, `integration_admin` |
| Token TTL | 15 min (access), 7 days (refresh) |
| Broker | OIDC/SAML для Ready4, PrimeZone, Подарок в квадрате |

**Данные в БД (`lkfl_keycloak`):**
- Users (связь с `lkfl_platform.users` по email)
- Realms, clients, roles, groups
- Sessions, authentication events
- Identity broker config (внешние провайдеры)

## Зависимости между пакетами

Кратко:
- **Frontend** → lkfl-server API (HTTP через Nginx, всё под `/api/v1/`)
- **`engagement/flow/`** → `billing.BillingService.Credit(ctx, ...)` — прямой Go call
- **`engagement/flow/`** → `integrationclient.Activate(ctx, ...)` — **gRPC** (localhost :8090) → proxy → провайдеры [M16]
- **`engagement/flow/`** → `payments.PaymentGateway.Authorize(ctx, ...)` — прямой Go call
- **`cel/`** → `llm.LLMClient.GenerateCel(ctx, ...)` — прямой Go call
- **`compliance/`** → `user/`, `engagement/flow/`, `notification/` — прямой Go call
- **`gamification/`** → Go-callback из `engagement/flow/` — `TriggerHandler.OnEngagementCompleted()`
- **`user/HRSync`** → HR-система напрямую (REST, Asynq worker)
- **`billing/payroll/`** → 1С напрямую (REST API)
- **lkfl-integration-proxy** → Провайдеры льгот (REST/SSO) — fault isolated, circuit breaker, credential store [M16]
- **Webhook** → провайдеры → Nginx `/webhooks/` → proxy :8091 → verify → монолит `/internal/webhook/callback` [M16]
- **Keycloak** → lkfl-server + Frontend (JWT validation), внешние сервисы (Identity Broker)

> **Важно:** между internal пакетами монолита — compile-time type safety через Go interfaces. Один `go build ./...` гарантирует корректность контрактов.
> **M16:** монолит ↔ proxy — gRPC (protoc генерация, один репозиторий, один go.mod).

## Multi-tenancy

Каждая бизнес-таблица содержит `tenant_id UUID REFERENCES tenants(id)`. Middleware `TenantResolver` извлекает tenant из JWT claim (`tenant_id`) или subdomain.

```sql
CREATE TABLE tenants (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    slug        VARCHAR(50) UNIQUE NOT NULL,
    brand_css_url TEXT NOT NULL DEFAULT '/theme/brand-default.css',
    created_at  TIMESTAMPTZ DEFAULT now()
);
```

## Релизная политика

> **M12:** один бинарник → один release.
> **M16:** два бинарника → независимый deploy proxy (адаптеры, circuit breaker) без redeploy монолита.

| Компонент | Релиз | Триггер |
|-----|-----|-|
| **lkfl-server** | Монолит | Новый endpoint, bugfix, биллинг правило, UI change |
| **lkfl-integration-proxy** | Независимо от монолита | Новый адаптер провайдера, изменение circuit breaker, webhook handler fix [M16] |
| **Frontend** | Независимо | Новый UI, багфикс, DS bump |

**Правило:** изменение API контракта (internal пакет → internal пакет монолита) → compile-time validation, один релиз монолита покрывает все изменения.
**M16:** изменение gRPC контракта (mono ↔ proxy) → protoc генерация, один go.mod, совместный build.
**ADR-021:** изменение CEL context schema → обновление `shared/pkg/celcontext` → все пакеты компилируются с одной версией.

> **M12:** Future split guidance — см. [ADR-024](./adr/024-modular-monolith.md). При split интерфейсы в internal пакетах заменяются на gRPC/NATS publisher без изменения HTTP-поверхности.
> **M16:** Integration Proxy — уже реализован как split для I/O boundary. [ADR-035](./adr/035-integration-proxy.md)
