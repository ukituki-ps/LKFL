Требования безопасности платформы гибких льгот, основанные на ФТ §5, 152-ФЗ, ФСТЭК, OWASP Top 10. Платформа обрабатывает персональные данные 100 000+ сотрудников.

## 1. OWASP Top 10 — защита от уязвимостей

| Уязвимость | Защита | Реализация |
|------------|--------|------------|
| **A01: Broken Access Control** | RBAC + tenant isolation | Middleware `TenantResolver`, claim `tenant_id` в JWT |
| **A02: Cryptographic Failures** | TLS 1.3 + AES-256-GCM | Nginx TLS, `golang.org/x/crypto` для ПДн |
| **A03: Injection** | Parameterized queries | `pgx` — только `$1, $2`, без string concat |
| **A04: Insecure Design** | Security by design | ADR 003 (Keycloak), ADR 005 (broker), consent check |
| **A05: Security Misconfiguration** | Hardened config | Nginx security headers, Docker non-root, least privilege |
| **A06: Vulnerable Components** | SAST/DAST scanning | `golangci-lint`, `trivy`, `npm audit`, CI pipeline |
| **A07: Auth Failures** | Keycloak OIDC | Мультифактор (опционально), password policy, session timeout |
| **A08: Data Integrity** | Signed webhooks, checksums | Webhook signature verification, XLSX hash |
| **A09: Logging Failures** | Structured JSON logs | Loki, audit trail для ПДн, correlation ID |
| **A10: SSRF** | URL allowlist | Валидация redirect URL для SSO провайдеров |

## 2. Шифрование данных

### При передаче

| Канал | Шифрование |
|-------|-----------|
| Client → Nginx | TLS 1.3 (min TLS 1.2) |
| Nginx → сервисы | Internal network (Docker bridge, no TLS required для dev) |
| Сервис → провайдер | TLS 1.2+ (mutual TLS где поддерживается) |
| Сервис → БД | `sslmode=disable` (dev) → `sslmode=verify-full` (prod) |
| Сервис → Redis | `REDIS_PASSWORD` + `requirepass` |

### При хранении

| Данные | Метод | Ключ | Шлюз |
|-------|-----|----|--|
| Пароли | bcrypt (cost 12) | Salt per password (bcrypt internal) | — |
| ПДн (адрес, телефон) | AES-256-GCM | Vault (prod) / env var (dev) | — |
| Согласия ПДн | Plain text + hash signature | SHA-256 для integrity check | — |
| Креденшиалы провайдеров | AES-256-GCM | Vault (prod) / env var (dev) | integrationclient/ (proxy) |
| Данные карт (пэйшлюз) | **Не хранятся** | Token-only от пэйшлюза | **M12:** `internal/payments/` (логическая изоляция в mono-бинарнике, PCI DSS → separation of concerns, ADR-018 → ADR-024) |

## 5. 152-ФЗ — обработка персональных данных

### Требования

| Требование | Реализация |
|------------|-----------|
| Данные не покидают РФ | Все серверы — РФ (Yandex Cloud / Selectel / on-premise) |
| Согласие на ПДн | Галка на регистрации + отдельный consent per провайдер |
| Отзыв согласия | API endpoint → `compliance.CascadeRevoke()` → каскадное удаление → audit log |
| Удаление по запросу | Worker `consent-revoke` → `compliance/` (M07 T0702): удаление из БД, нотификация, ФСТЭК-логирование |
| Дети до 18 лет | Parental consent workflow (родительский email подтверждение) |
| Согласие родственников 18+ | SMS-код подтверждения |

### Consent lifecycle

```
Создание → Действие → Продление → Отзыв → Удаление

1. Регистрация → согласие на обработку (обязательно)
2. Подключение льготы → согласие на передачу данных провайдеру (опционально per provider)
3. Через 12 мес → напомнить о продлении согласия
4. Письменный отзыв → API revoke → каскадное удаление
5. Удаление → audit log → подтверждение HR
```

### Consent store

> Каноническая схема таблицы `consents` — [`архитектура/schema.md §3.2`](./schema.md#consents-s01-s02-s10-j13c-j14a-j14b).
> Ниже — краткое резюме полей, релевантных для 152-ФЗ:

| Поле | Назначение | 152-ФЗ |
|------|-----------|--------|
| `tenant_id` | Multi-tenancy изоляция | ✅ |
| `provider` | Конкретный провайдер (NULL = общее) | ✅ per-provider согласие |
| `scope` | Описание согласия | ✅ текст согласия |
| `granted_at` | Время дачи согласия | ✅ |
| `revoked_at` | Время отзыва | ✅ |
| `method` | `checkbox` / `sms_code` / `parental` | ✅ способ подтверждения |
| `ip_address` | IP при согласии | ✅ audit trail |
| `user_agent` | Browser UA при согласии | ✅ audit trail |
| `content_hash` | SHA-256 текста согласия | ✅ integrity check |
| `audit_log` | JSONB история изменений | ✅ ФСТЭК logging |

## 6. ФСТЭК — сертификация

### Требования платформы

| Требование | Реализация |
|------------|-----------|
| Единое место хранения | Все БД на одном кластере PostgreSQL (РФ) |
| Audit trail | Структурированные логи всех операций с ПДн |
| Разграничение доступа | RBAC: сотрудник → свои данные, HR → все данные tenant, admin →infra |
| Защита от несанкционированного доступа | Keycloak auth, middleware, rate limiting |
| Резервное копирование | PostgreSQL WAL archiving + pg_basebackup |
| Мониторинг | Prometheus + Grafana + alerting |

### Процесс сертификации

1. **Подготовка документации:** описание архитектуры (этот раздел), политики безопасности, threat model
2. **Penetration testing:** внешний аудит
3. **Исправление уязвимостей:** по результатам пентеста
4. **Повторный пентест:** подтверждение исправлений
5. **Подача заявки:** в аккредитованный центр ФСТЭК
6. **Экспертиза:** проверка документации + кода
7. **Сертификат:** срок действия 3 года

**Срок:** начать подготовку минимум за 6 месяцев до запуска (лето 2026).

## 7. Rate limiting

| Endpoint | Лимит | Окно |
|----------|-------|------|
| Auth (login) | 10 req | 1 мин |
| Auth (register) | 5 req | 1 мин |
| Auth (password reset) | 3 req | 15 мин |
| API (общий) | 100 req | 1 мин |
| Billing (balance) | 30 req | 1 мин |
| Webhook receiver | 1000 req | 1 мин |

**Per-tenant изоляция:** ключ rate limiter'a включает `tenant_id`:
```
rate_limit:{tenant_id}:{user_id}:{endpoint}
```
Один tenant не может затопить систему за счёт другого.

Реализация: Redis sliding window (`INCR` + `EXPIRE`), middleware.

## 8. Audit trail

Все операции с ПДн и финансами логируются:

```json
{
  "ts": "2026-05-22T12:00:00Z",
  "level": "info",
  "svc": "platform",
  "action": "consent.granted",
  "user_id": "uuid",
  "tenant_id": "sdek",
  "provider": "alpha",
  "ip": "192.168.1.1",
  "ua": "Mozilla/5.0..."
}
```

| Событие | Логирование |
|---------|-------------|
| Регистрация | user_id, email, IP, UA, tenant_id |
| Вход | user_id, IP, UA, success/fail |
| Изменение профиля | user_id, field, old_value, new_value |
| Согласие (grant/revoke) | user_id, provider, scope, method, IP |
| Транзакция биллинга | user_id, type, amount, source, period_id |
| Подключение энгейджмента | user_id, engagement_id, provider, status |
| Загрузка XLSX | admin_id, file_hash, records_count, errors |
| Генерация документа | user_id, doc_type, storage_path |

## 9. Nginx security headers

```nginx
add_header X-Content-Type-Options nosniff;
add_header X-Frame-Options DENY;
add_header X-XSS-Protection "0";  # handled by CSP
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'sha256-<HASH>'; style-src 'self' 'sha256-<HASH>' fonts.googleapis.com; font-src fonts.gstatic.com; img-src 'self' data:; connect-src 'self' *.sentry.io;";
add_header Referrer-Policy strict-origin-when-cross-origin;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()";
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

> **Примечание:** `'sha256-<HASH>'` — заглушка. В production подставить реальные SRI-хэши для inline-скриптов и стилей. `unpkg.com` удалён — публичный CDN не допускается в production (supply chain risk). Для dev-среды допустимо `'unsafe-inline'` с отключением в CI.

## 13. Threat Model

### STRIDE-анализ

| Threat | STRIDE | Вектор | Митигация |
|--------|--------|--------|-----------|
| **Подделка JWT** | Spoofing | Атакующий создаёт поддельный токен | Keycloak JWT verification (signature check), short TTL (15 min) |
| **Меж-tenant доступ** | Spoofing | User tenant A запрашивает данные tenant B | `tenant_id` middleware isolation, FK constraint, Keycloak realm per tenant |
| **SQL Injection** | Tampering | Ввод SQL-кода в формах | pgx parameterized queries (`$1, $2`), без string concat |
| **XSS** | Tampering | Скрипт в textarea/subject | CSP headers, React auto-escaping, input validation |
| **CSRF** | Repudiation | Фальшивая форма на другом сайте | SameSite cookies, anti-CSRF tokens |
| **UAC (Privilege Escalation)** | Repudiation | Employee получает доступ HR | RBAC middleware, Keycloak role enforcement, audit trail |
| **Данные в логах** | Information Disclosure | ПДн в JSON-логах | ПДн не логируется (только user_id), AES-256-GCM для ПДн в БД |
| **SSRF** | Repudiation | Redirect URL к внутренним ресурсам | URL allowlist для провайдеров, валидация redirect_url |
| **DoS** | Denial of Service | Flood запросов | Rate limiting per tenant, Redis sliding window |
| **Утечка через API** | Information Disclosure | Чрезмерная детализация в API | Response filtering, no internal fields in API responses |

### Kill Chain — сценарии

#### S1: Compromise JWT → доступ к чужим данным

```
Attacker → Подделывает JWT → Bypass tenant isolation → Чтение данных другого tenant'а
```

**Митигация:**
1. Keycloak signature verification (RS256)
2. JWT TTL 15 min — минимизация окна атаки
3. `TenantResolver` middleware — проверяет claim `tenant_id` + DB FK
4. Audit log — все запросы логируются (user_id, tenant_id, endpoint)

#### S2: Mass data exfiltration через HR-аккаунт

```
Attacker → Компрометирует HR-аккаунт → Массовый экспорт ПДн → Утечка
```

**Митигация:**
1. HR аккаунт — Keycloak MFA required
2. Rate limiting на export endpoint
3. Audit log — все операции HR логируются (action, resource, ip)
4. ФСТЭК — audit trail immutable, DELETE не разрешён

#### S3: Billing manipulation

```
Attacker → Подделывает billing event → Credit себе → Увеличение баланса
```

**Митигация:**
1. Billing events — только из FlowEngine (internal call, не из API)
2. `idempotency_key` — защита от дублирования
3. ACID — одна transaction для engagement + billing
4. Audit trail — все транзакции логируются с контекстом

## 14. Integration Proxy — credential isolation

> **Связь с ADR-035:** M16, вынос внешних интеграций из монолита в отдельный бинарник `lkfl-integration-proxy`.

### Credential isolation

Провайдерские API-креденшиалы (API-ключи, токены, сертификаты) хранятся **исключительно** в Integration Proxy — в таблице `providers` схемы `lkfl_integration`.

| Где | Что хранится |
|-----|-------------|
| **Proxy** (`lkfl-integration-proxy`) | Provider API credentials (encrypted in `providers` table, decrypted at runtime) |
| **Монолит** (`lkfl-server`) | Ничего — не знает о креденшиалах провайдеров |
| **HashiCorp Vault** | Master key для decryption (production) |
| **Environment variables** | Master key (development, `INTEGRATION_MASTER_KEY`) |

**Безопасность при компрометации:** если монолит скомпрометирован, креденшиалы провайдеров остаются в безопасности — они физически изолированы в другом бинарнике и другой DB schema.

### Шифрование credentials

- **Алгоритм:** AES-256-GCM для полей `credentials_encrypted` в таблице `providers` (schema `lkfl_integration`)
- **Master key для decryption:**
  - **Production:** HashiCorp Vault (secrets engine, динамическая ротация)
  - **Development:** environment variable `INTEGRATION_MASTER_KEY` (не коммитить, не передавать в CI-логи)
- **Decrypt на runtime:** credentials decrypt'ятся при старте proxy и хранятся в памяти (не на диске)

### Webhook attack surface

Webhook endpoint'ы живут **только** на Integration Proxy (порт :8091 HTTP), не на монолите (:8080).

```
Внешний провайдер → Nginx /webhooks/:provider → Proxy (:8091)
                                                            │
                                                  1. Verify signature
                                                  2. Validate payload
                                                  3. Store event in DB (webhook_events)
                                                  4. HTTP POST /internal/webhook/callback → Монолит (:8080)
```

**Rate limiting на webhook endpoint** — на уровне Nginx (см. §7 Rate limiting, `Webhook receiver: 1000 req/1 мин`).

**Монолит не exposes публичных webhook endpoint'ов.** Внутренний endpoint `/internal/webhook/callback` доступен только из Docker-сети (не через Nginx).

### gRPC security

Коммуникация между монолитом и proxy — gRPC на localhost.

| Среда | TLS | Обоснование |
|-------|-----|------------|
| **Development / Docker Compose** | Нет (plaintext gRPC) | Same host, Docker bridge network, нет внешнего доступа |
| **Production (отдельные хосты)** | **mTLS** | Разные хосты — требуется mutual TLS для аутентификации обоих сторон |

gRPC порт: 8090. HTTP webhook порт: 8091. Оба порта не экспонируются за пределы Docker-сети.

### STRIDE update — добавлен proxy component

| Threat | STRIDE | Вектор | Митигация |
|--------|--------|--------|-----------|
| **Proxy compromise** | Information Disclosure | Атакующий получает доступ к proxy → креденшиалы провайдеров | AES-256-GCM для credentials, master key в Vault, доступ к Vault — отдельный IAM policy |
| **Mono compromise** | Information Disclosure | Атакующий получает доступ к монолиту → бизнес-данные, но **не** креденшиалы провайдеров | Credential isolation: провайдерские ключи только в proxy (таблица `providers` в schema `lkfl_integration`) |
| **Webhook injection** | Tampering | Фейковый webhook от атакующего → подделка события | Signature verification на proxy, rate limiting на Nginx, payload validation |
| **gRPC MITM** | Spoofing | Man-in-the-middle между mono и proxy | mTLS в production (отдельные хосты); localhost — нет внешнего доступа в Docker |
| **Credential exfiltration** | Information Disclosure | Dump DB proxy → encrypted credentials → brute force | AES-256-GCM (не обратим без master key), Vault access control, audit log на decryption |

### Сценарии компрометации

#### S4: Компрометация Integration Proxy

```
Attacker → Компрометирует proxy → Доступ к credentials_encrypted (зашифрованы)
→ Без master key → credentials не расшифровываются → доступ к провайдерам заблокирован
```

**Митигация:**
1. AES-256-GCM — без master key данные не расшифровать
2. Master key в Vault — отдельный IAM policy, не в docker-compose
3. Audit log на операции decryption
4. Если master key скомпрометирован → ротация всех provider credentials + новый master key

#### S5: Компрометация монолита

```
Attacker → Компрометирует lkfl-server → Доступ к бизнес-данным (users, engagements, billing)
→ Попытка доступа к provider credentials → НЕТ ДАННЫХ (credential isolation)
```

**Митигация:**
1. Credential isolation: mono не хранит provider credentials
2. gRPC client в mono — только вызывает proxy, не получает credentials
3. Монолит может быть скомпрометирован без риска для провайдерских интеграций

#### S6: Webhook injection attack

```
Attacker → Фейковый webhook (имитация провайдера) → Proxy
→ Signature verification → FAIL → reject → audit log
```

**Митигация:**
1. Signature verification (HMAC-SHA256) на каждом webhook
2. Payload validation (schema check)
3. Rate limiting на Nginx (1000 req/1 мин)
4. Все webhook события логируются в `webhook_events` (audit trail)
